# QA Report — E02S02: Parse SMS Reply, Match Tech, Mark Job Assigned

## Verdict: PASS

All 6 acceptance criteria met. No rule violations. No remediation cycles required.

---

## Per-Criterion Results

### AC1 — n8n extracts tech name from SMS body (case-insensitive match)

**Result: PASS**

Evidence:
- `Extract Tech Name` Code node: `rawText.trim()` from `$('Parse SMS Fields').item.json.message_body`
- `Get Tech by Name` HTTP Request: `name=ilike.{{ $('Extract Tech Name').item.json.tech_name }}`
- Supabase `ilike` operator provides case-insensitive matching server-side
- "Mike", "mike", "MIKE" all resolve identically
- n8n v2 expression pattern `$('Node Name').item.json.field` used correctly throughout

### AC2 — Query techs filtered by client_id — match scoped to that client only

**Result: PASS**

Evidence:
- `Get Client by Phone` GET: `clients?dispatcher_phone=eq.[from_phone]` → yields `client_id`
- `Get Tech by Name` GET: `techs?client_id=eq.[client_id]&name=ilike.[tech_name]`
- Both filters applied in one query — structural prevention of cross-client access
- `Client Found?` IF node: exits silently if `$('Get Client by Phone').item.json.id` is empty
- n8n v2 direct element access pattern used correctly (no `$json[0]` anti-pattern)

### AC3 — On match, PATCH jobs: tech_id, status=Assigned, assigned_at=now()

**Result: PASS**

Evidence:
- `Assign Job` HTTP Request: PATCH `/rest/v1/jobs?id=eq.[job_id]`
- Body sets all three required fields:
  - `tech_id`: `$('Get Tech by Name').item.json.id`
  - `status`: `"Assigned"` (literal)
  - `assigned_at`: `$now.toISO()` (ISO 8601 current timestamp)
- `Prefer: return=representation` header included — updated row returned
- PATCH only fires after both `Tech Found?` (true) and `Job is New?` (true) guards pass
- HTTP Request node used — not native Supabase node (AC6 compliant)

### AC4 — Confirmation SMS "Job #[ID] assigned to [Tech Name]." sent to dispatcher

**Result: PASS**

Evidence:
- `Build Confirmation SMS` Code node: `` `Job #${jobShort} assigned to ${techName}.` ``
- `jobShort`: `$('Get Latest New Job').item.json.id.slice(-6).toUpperCase()` — correct Job # format
- TO: `$('Parse SMS Fields').item.json.from_phone` — the dispatcher's phone (sender of original SMS)
- FROM: `$('Get Client by Phone').item.json.twilio_number` — client's registered Twilio number
- `Send Confirmation SMS`: `n8n-nodes-base.twilio`, typeVersion 1 — native Twilio node ✅
- `continueOnFail: true` — SMS failure does not revert the assignment
- Template matches conventions.md "Job # format" convention exactly

### AC5 — If job status is not New, assignment does not proceed

**Result: PASS**

Evidence:
- `Get Latest New Job` GET: `status=eq.New` filter in Supabase URL — only New jobs returned
- `Job is New?` IF node: `$('Get Latest New Job').item.json.status` equals `"New"` (strict)
- If no New job: empty Supabase response → status field is undefined → condition fails → exits cleanly
- If job has any other status (Assigned/En Route/On Site/Complete): condition fails → exits cleanly
- False branch is unwired → silent exit — E02S04 will add guard SMS at this exact hook point

### AC6 — All Supabase queries use HTTP Request nodes — not native Supabase nodes

**Result: PASS**

All Supabase node audit:

| Node | Type | Version |
|------|------|---------|
| Get Client by Phone | n8n-nodes-base.httpRequest | 4.2 |
| Get Tech by Name | n8n-nodes-base.httpRequest | 4.2 |
| Get Latest New Job | n8n-nodes-base.httpRequest | 4.2 |
| Assign Job | n8n-nodes-base.httpRequest | 4.2 |

Zero instances of `n8n-nodes-base.supabase` or any native Supabase node type found in the workflow JSON.
