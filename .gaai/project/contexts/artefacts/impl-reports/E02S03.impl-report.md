## Acceptance Criteria Coverage

### AC1 — Error SMS sent to dispatcher phone when no tech match found

**Status: Implemented**

The `Tech Found?` IF node's FALSE output (index 1) now routes to the error SMS branch.
The `Send Error SMS` native Twilio node uses `to = $('Build Error SMS').item.json.dispatcher_phone`,
which is sourced from `$('Parse SMS Fields').item.json.from_phone` — the dispatcher's own number.

### AC2 — Error SMS lists valid tech names and identifies the unrecognized reply

**Status: Implemented**

The `Build Error SMS` Code node constructs:
```
`No match for '${attempted}'. Valid techs: ${techNames}. Reply with a name to assign.`
```

`attempted` = `$('Extract Tech Name').item.json.tech_name` (what the dispatcher sent)
`techNames` = all names from the `Fetch All Techs` response, joined by `', '`

This matches the exact format specified in the story: `"No match for '[reply]'. Valid techs: Mike, Jake, Sara. Reply with a name to assign."`

### AC3 — Error SMS sent from same Twilio number (clients.twilio_number)

**Status: Implemented**

`from_number = $('Get Client by Phone').item.json.twilio_number` is captured in `Build Error SMS`
and passed to `Send Error SMS` `from` field. The number is never hardcoded.

### AC4 — Job status remains unchanged (still New)

**Status: Implemented**

The FALSE branch of `Tech Found?` contains no PATCH node and no call to the jobs table.
The only nodes are: Fetch All Techs (GET techs) → Build Error SMS (Code) → Send Error SMS (Twilio).
The job record is structurally unreachable from this path.

### AC5 — Valid tech list pulled live from techs table at error time

**Status: Implemented**

The `Fetch All Techs` HTTP Request node queries Supabase at runtime:
```
GET /rest/v1/techs?client_id=eq.[client_id]&select=name
```

The response is an array of `{ name }` objects. The `Build Error SMS` Code node uses
`$input.all().map(item => item.json.name).filter(Boolean).join(', ')` to build the list.
This is a live query — if techs are added or removed between SMSes, the error message
always reflects the current roster.

## Rules Applied

- HTTP Request nodes for all Supabase calls (conventions.md)
- Native Twilio node for SMS (story notes: NOT HTTP Request)
- n8n v2 expressions: `$('NodeName').item.json.field` throughout
- `$input.all()` used in Build Error SMS to iterate the multi-row techs response
- Phone numbers passed directly to Twilio node params (no URL encoding needed)
- `continueOnFail: true` on Twilio node — SMS failure does not affect job state
- Version ID bumped: `e02s01-s03-combined-v1`

---

## Known Risks / Limitations

1. **SUPABASE_ANON_KEY_PLACEHOLDER** — 5 HTTP Request nodes (was 4) now need the real anon key.
2. **Twilio credential ID** — Both Twilio nodes share the same credential reference `twilio-credentials`.
3. **Tech FALSE branch re-wiring** — Per known n8n v2 import behavior, manually verify after import.
4. **SMS length** — Message stays under 160 chars for typical rosters (≤6 short names). With many
   techs or long names it may spill to 2 segments — acceptable behavior, not a functional defect.
