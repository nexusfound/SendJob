# E02S07 Implementation Report — Dashboard: Dropdown Assignment

## Summary

- Added tech dropdown to every `New`-status job row in the dispatcher dashboard, populated from the Supabase `techs` table filtered by `client_id`
- Implemented `assignTech()` PATCH function matching the SMS assignment path (identical fields: `tech_id`, `status = Assigned`, `assigned_at`)
- Row updates in-place on success (no page reload); inline error shown on failure; non-New rows render read-only tech name

## Rules Applied

- Design system: `--navy-mid` background, `--border` border, `--text` color, 16px font-size, 52px min-height (design-system.md)
- `escapeHtml()` applied to all user-sourced data in innerHTML
- `encodeURIComponent()` on URL path parameters
- Double-submit guard via `selectEl.disabled = true` before async call
- Graceful degradation when `CLIENT_ID` is placeholder or `loadTechs` fails
- No new files created; single-file modification

## Known Limitations

- `CLIENT_ID` must be replaced with the real client UUID before deployment. Currently set to `'REPLACE_WITH_YOUR_CLIENT_UUID'` — documented in config comment.
- `loadTechs()` and `loadJobs()` run concurrently on page load. If techs resolve after jobs render, the dropdown will show "No techs loaded" until next page load. This is a known race condition noted in the execution plan; the guard handles it gracefully (dropdown disabled, not broken). For production, consider awaiting `loadTechs` before `loadJobs`, or adding a refresh button.
