# Implementation Report — E02S08: Dispatcher Login (Supabase Auth)

## Summary

- Added Supabase Auth JS v2 CDN script and `createClient` initialization to `dispatcher-dashboard/index.html`
- Added full login form UI gating the dashboard: email/password inputs with 16px font and 52px tap targets per AC7 mobile requirements
- Implemented `getSession()` on page load (AC5), `signInWithPassword` on form submit (AC2/AC3/AC4), and `signOut` on logout button (AC6)
- Threaded the authenticated user JWT (`currentAccessToken`) into all three Supabase REST fetch calls (loadJobs, loadTechs, assignTech) to satisfy RLS authenticated policies (AC8)

## Rules Applied

- Design system: `--navy-mid` background, `--border` borders, `--ice` focus ring, Barlow font family, `--tap-min: 52px` — all per `context.md` Playbook design system
- No new files created; single-file modification within story scope
- Error handling present on all async operations (auth calls, fetch calls)
- `novalidate` on form with explicit JS validation — prevents native browser validation from interfering with custom error display

## RLS Note (AC8)

The `Authorization: Bearer {access_token}` header is now sent on all Supabase REST calls. Existing Sprint 1 `anon` policies remain intact. If those policies are removed or scoped to `anon` only, add `auth.role() = 'authenticated'` policies in Supabase Dashboard for `jobs` and `techs` tables. Dispatcher account must exist in Supabase Auth before testing.

## Known Risks / Decisions

- Session storage: Supabase JS v2 uses `localStorage` by default — session persists across page refreshes without additional code. This satisfies AC5 automatically.
- `status-dot` and `status-label` are cleared in `showLoginView()` since no data is loaded on the login screen. They are re-animated by `showLoading()` when `loadJobs()` runs after login.
- No friction — implementation was straightforward, no retries required.
