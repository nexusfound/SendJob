# E03S03 Implementation Report — N8N: Tech Notification SMS with Magic Link

## Summary

- Created `n8n-workflows/E03S03-tech-notification-sms.json` — an 8-node importable n8n workflow that, triggered by a POST webhook from E03S02, fetches job/client/tech data from Supabase, assembles the canonical tech notification SMS, and sends it via the native n8n Twilio node. An IF branch handles the email path as a logged stub per AC6.
- All 3 Supabase reads use HTTP Request nodes (typeVersion 4.2) per DEC-003. The Twilio send uses the native Twilio node (NOT HTTP Request) per AC7 and conventions. No `require('crypto')` used — token arrives via webhook payload from E03S02.
- Business name is always fetched from `clients.business_name` (never hardcoded). SMS template is canonical. Magic link URL is `https://tech.sendjob.app?token=[token]`. Opt-out language appended on every message for A2P 10DLC compliance.

## Rules Applied

- DEC-003: All Supabase access via HTTP Request node (typeVersion 4.2) — no native Supabase nodes
- conventions.md / AC7: Twilio SMS via native n8n Twilio node (NOT HTTP Request)
- conventions.md: `$('Node Name').item.json.field` pattern used for all upstream references — never `$json.field`
- conventions.md: `return [{ json: { ... } }]` in all Code node outputs
- IMPORTANT CONTEXT: No `require('crypto')` in any Code node — token is passed from E03S02 via webhook payload
- IMPORTANT CONTEXT: HTTP Request typeVersion 4.2, Code typeVersion 2, Webhook typeVersion 2
- No credentials embedded: all SUPABASE_ANON_KEY_PLACEHOLDER with comment directing replacement
- Opt-out language appended per Twilio A2P 10DLC compliance requirements and story notes
- Magic link URL: `https://tech.sendjob.app?token=[token]` per AC3 (story authoritative over conventions URL variant)

## Architecture Note — Token Flow

Token is NOT generated by this workflow. It is generated and stored by E03S02 (`"Generate Magic Link Token"` node). E03S03 receives the token via the webhook POST body (`$('Trigger: Tech SMS Webhook').item.json.body.token`). This keeps E03S03 stateless with respect to token generation and avoids any dependency on `require('crypto')`.

Integration is via webhook: E03S02's `Token PATCH Succeeded?` true branch calls `POST /webhook/sendjob-tech-sms` with the token payload. The `_comment` field in the JSON contains precise integration instructions referencing actual E03S02 node names.

## Known Limitations / Deployment Notes

- Replace `SUPABASE_ANON_KEY_PLACEHOLDER` in 3 nodes (GET Job Details, GET Client Details, GET Tech Details) before importing
- Twilio credential: configure in n8n Settings → Credentials → New → Twilio (Account SID + Auth Token); update credential ID in `Send Tech SMS` node to match
- RLS policies required: `anon` SELECT on `jobs`, `clients`, `techs` tables
- E03S02 integration: see `_comment` in JSON for exact HTTP Request node configuration to wire E03S02 → E03S03
- `token_issued_at` in E03S02 uses `$now.toISO()` — if that fails in n8n 2.6.4, fallback is `new Date().toISOString()` (noted in E03S02 report; not a concern for E03S03)
- Test on actual mobile device per story notes
