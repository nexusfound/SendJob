|------|---------|-----|

## Acceptance Criteria Coverage

### AC1 — n8n extracts tech name from SMS body (case-insensitive match)

**Status: Implemented**

The `Extract Tech Name` Code node trims the `message_body` field from `Parse SMS Fields`.
The `Get Tech by Name` HTTP Request node uses Supabase's `ilike` operator for case-insensitive
matching: `techs?name=ilike.[tech_name]`. A dispatcher sending "Mike", "mike", or "MIKE" will
all resolve to the same tech record.

Expression access pattern (n8n v2):
```
$('Extract Tech Name').item.json.tech_name  // trimmed dispatcher reply text
```

### AC2 — Query techs table filtered by client_id — match scoped to that client only

**Status: Implemented**

The `Get Client by Phone` HTTP Request node looks up the client record using
`dispatcher_phone=eq.[from_phone]`. The resulting `client_id` is passed to
`Get Tech by Name`: `techs?client_id=eq.[client_id]&name=ilike.[tech_name]`.

Both filters are applied simultaneously — a tech named "Mike" at client A will never
match a tech named "Mike" at client B. Cross-client contamination is structurally
prevented by the `client_id` scope.

The `Client Found?` IF node guards against unknown dispatcher phone numbers — if no
client record exists for the incoming `from_phone`, the workflow exits silently.

### AC3 — On successful match, PATCH jobs: tech_id, status=Assigned, assigned_at=now()

**Status: Implemented**

The `Assign Job` HTTP Request node sends a PATCH to:
`/rest/v1/jobs?id=eq.[job_id]`

With body:
```json
{
  "tech_id": "{{ $('Get Tech by Name').item.json.id }}",
  "status": "Assigned",
  "assigned_at": "{{ $now.toISO() }}"
}
```

`Prefer: return=representation` header is set — Supabase returns the updated row.
This confirms the PATCH was applied and provides the updated record for downstream use
(E03S02 magic link token generation).

The `Tech Found?` IF node ensures the PATCH only fires when a matching tech exists.
The `Job is New?` IF node ensures the PATCH only fires when a New job is available.

### AC4 — After successful assignment, dispatcher receives confirmation SMS

**Status: Implemented**

The `Build Confirmation SMS` Code node assembles:
```
Job #[LAST_6_UUID_UPPERCASE] assigned to [Tech Name].
```

Job # format: `jobId.slice(-6).toUpperCase()` — consistent with conventions.md job ID format.

The `Send Confirmation SMS` native Twilio node sends to `dispatcher_phone` (from `from_phone`)
from `clients.twilio_number` (never hardcoded — pulled from the client record).

The native n8n Twilio node is used — NOT an HTTP Request node (per conventions.md).
`continueOnFail: true` — SMS failure does not revert the job assignment.

### AC5 — If job status is not New, assignment does not proceed

**Status: Implemented**

The `Get Latest New Job` HTTP Request node queries:
`jobs?client_id=eq.[id]&status=eq.New&order=created_at.desc&limit=1`

The Supabase `status=eq.New` filter means only New jobs are returned. The `Job is New?` IF
node checks `$('Get Latest New Job').item.json.status === 'New'`:

- If the most recent job has any other status (Assigned, En Route, On Site, Complete): the
  status field won't be "New" → IF exits on false branch → no PATCH, no SMS
- If no New job exists at all: the response will be an empty array; the status field will
  be undefined → IF condition fails → exits cleanly

This cleanly separates E02S04 (already-assigned guard SMS) — the false branch of `Job is New?`
is the exact hook point where E02S04 will add its guard SMS node.

### AC6 — All Supabase queries use HTTP Request nodes — not native Supabase nodes

**Status: Implemented**

All four Supabase interactions use HTTP Request nodes (typeVersion 4.2):
- `Get Client by Phone`: GET clients
- `Get Tech by Name`: GET techs
- `Get Latest New Job`: GET jobs
- `Assign Job`: PATCH jobs

Native n8n Twilio node is used only for SMS (correct — Twilio uses native node, per conventions).
No native Supabase nodes appear anywhere in the workflow.

## Rules Applied

- HTTP Request nodes only for all Supabase calls (GET + PATCH) — no native Supabase nodes
- Native n8n Twilio node for SMS confirmation — not HTTP Request
- n8n v2 expression pattern: `$('Node Name').item.json.field` throughout
- Code node output: `return [{ json: { ... } }]` in both Code nodes
- `Prefer: return=representation` on PATCH to get updated row back
- Supabase anon key: placeholder convention (`SUPABASE_ANON_KEY_PLACEHOLDER`) — human replaces
- `continueOnFail: true` on Twilio SMS node — SMS failure does not revert assignment
- Job # format: `.slice(-6).toUpperCase()` per conventions.md naming conventions

## Known Risks / Limitations

1. **SUPABASE_ANON_KEY_PLACEHOLDER**: Four HTTP Request nodes require the real anon key before activation.
2. **Twilio credential ID**: The `Send Confirmation SMS` node has `id: "twilio-credentials"` — this
   must be updated to match the actual credential ID in the n8n instance.
3. **A2P campaign status**: Twilio A2P 10DLC may still be pending. The `from_number` is pulled from
   `clients.twilio_number` — this number must be approved for A2P before live sends succeed.
4. **Multiple New jobs**: If a client has multiple New jobs, `order=created_at.desc&limit=1` assigns
   the most recent one. This is intentional — the dispatcher's reply is assumed to be for the latest
   unassigned job.
5. **Tech name partial match**: `ilike` in Supabase matches the entire name field against the entire
   SMS reply text. If the dispatcher sends "Tech Mike" instead of "Mike", the match will fail.
   This is correct behavior — the dispatcher should send just the tech name.
6. **E02S01 standalone vs Option B**: If the workflow is imported standalone (Option A), the webhook
   path `sendjob-intake` is shared with E02S01. Verify that n8n handles multiple active workflows
   on the same webhook path — behavior may vary by n8n version. Option B is safer.
