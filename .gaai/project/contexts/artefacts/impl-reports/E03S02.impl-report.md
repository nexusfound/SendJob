# E03S02 Implementation Report — N8N: Magic Link Token on Job Assignment

## Summary

- Added `generateMagicLinkToken(techId)` function to the dispatcher dashboard to generate a UUID token and PATCH `techs.magic_link_token` + `techs.token_issued_at` after every dashboard assignment — fire-and-forget, never affects job assignment state on failure
- Created `n8n-workflows/E03S02-magic-link-token.json` — importable n8n workflow (4-node sequence: Code → HTTP PATCH → IF check → Log Error) for the E02S02 SMS assignment path integration
- Token generation satisfies both assignment paths (AC4) using the architecture already proven in the dashboard (browser fetch) and n8n (HTTP Request node per DEC-003)

## Files Changed

| File | Action | Purpose |
|------|--------|---------|
| `dispatcher-dashboard/index.html` | Modified | Added `generateMagicLinkToken()` function + one-line call in `assignTech()` success handler |
| `n8n-workflows/E03S02-magic-link-token.json` | Created | n8n workflow for SMS assignment path — importable, documented, two integration modes |

## Acceptance Criteria Mapping

| AC | Change | Location |
|----|--------|----------|
| AC1 | `crypto.randomUUID()` with fallback for old browsers | `generateMagicLinkToken()` in `index.html` / Code node in `E03S02-magic-link-token.json` |
| AC2 | PATCH `/rest/v1/techs?id=eq.[techId]` with `{ magic_link_token, token_issued_at }`, `Prefer: return=representation` | `generateMagicLinkToken()` fetch body / `PATCH techs — Store Token` HTTP Request node |

| AC4 | Dashboard: `generateMagicLinkToken(techId)` in `assignTech()` success handler. n8n SMS path: `E03S02-magic-link-token.json` workflow | `index.html` line in `assignTech()` / n8n workflow JSON |
| AC5 | Dashboard: token `.catch()` logs via `console.warn()` — does not call `showAssignError()`, does not re-enable select, does not affect prior job PATCH. n8n: `continueOnFail: true` + false branch logs only, no `jobs` table writes | `generateMagicLinkToken()` catch block / `PATCH techs — Store Token` continueOnFail setting |

## Rules Applied

- DEC-003: HTTP Request node for Supabase in n8n; `fetch()` in browser dashboard — no native Supabase nodes
- DEC-013: Token stored in `techs.magic_link_token` and `techs.token_issued_at` — per-tech, not per-job
- `Prefer: return=representation` on all Supabase PATCH calls
- `encodeURIComponent()` on URL path parameter (`techId`)
- `$('Node Name').item.json.field` reference pattern in all n8n expressions
- `return [{ json: { ... } }]` in all Code node outputs
- `continueOnFail: true` on HTTP Request node to implement AC5 in n8n workflow
- No secrets embedded: n8n JSON uses `SUPABASE_SERVICE_ROLE_KEY_OR_ANON_KEY` placeholder with explicit instructions

## Architecture Note — AC4 Dual-Path

The E02S07 assignment path is entirely browser-side (no n8n). Token generation for this path is implemented directly in `dispatcher-dashboard/index.html` using `crypto.randomUUID()` and the same Supabase PATCH pattern as the dashboard's existing writes (DEC-014). This is architecturally consistent and satisfies AC4 without scope change.

The E02S02 SMS path uses the n8n workflow JSON (`E03S02-magic-link-token.json`). The JSON includes two integration modes: standalone webhook or inline node sequence appended to the E02S02 workflow. E02S02 is not yet delivered — the n8n workflow is the ready-to-integrate artefact for when E02S02 is built.

## Known Limitations / Deployment Notes

- `n8n-workflows/E03S02-magic-link-token.json`: Replace `SUPABASE_SERVICE_ROLE_KEY_OR_ANON_KEY` with the real key before importing. Service role key recommended (bypasses RLS). If using anon key, verify `techs` table has anon UPDATE policy in Supabase.
- E02S02 integration: The n8n JSON `_comment` field documents the exact steps to inline the token nodes into the E02S02 workflow when it is built. The `tech_id` reference must use the correct upstream node name from E02S02.
- `token_issued_at` uses `$now.toISO()` in n8n expressions — if this fails in n8n 2.6.4, fallback to `={{ new Date().toISOString() }}`.
