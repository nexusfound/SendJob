# Implementation Report — E02S04: Already-Assigned Guard SMS

## Summary of Changes

| File | Purpose | AC |
|------|---------|-----|
| `n8n-workflows/sendjob-intake-combined.json` | Added 3 nodes on false branch of `Job is New?`; updated `_NOTES` + `versionId` | AC1–AC5 |

Node count: 16 (before) → 19 (after). Three new nodes added: `Get Current Job State`, `Build Guard SMS`, `Send Guard SMS`.

---

## Acceptance Criteria Coverage

### AC1 — Before attempting to assign a tech, n8n checks `jobs.status` for the relevant job

**Status: Implemented**

The pre-existing `Job is New?` IF node (E02S02) checks `$('Get Latest New Job').item.json.status === 'New'` before any assignment. This gates the entire assignment path. E02S04 hooks into the false branch of this node — when `Job is New?` fires FALSE, `Get Current Job State` fetches the actual job state (without the `status=eq.New` filter) to build the informational guard message.

### AC2 — If `jobs.status` is not `New`, no assignment is made — the job record is not modified

**Status: Implemented**

The false branch of `Job is New?` leads exclusively to `Get Current Job State` → `Build Guard SMS` → `Send Guard SMS`. No PATCH node exists on this branch. The jobs table is never written. The guard SMS is purely informational.

### AC3 — Dispatcher receives SMS: `"Job #[ID] is already [status]. Assigned to [Tech Name]."`

**Status: Implemented**

The `Build Guard SMS` Code node assembles:
```javascript
const jobShort = job.id.slice(-6).toUpperCase();
const currentStatus = job.status || 'Unknown';
// With tech:
guardMsg = `Job #${jobShort} is already ${currentStatus}. Assigned to ${techName}.`;
```

`Send Guard SMS` is a native Twilio node that delivers this to the dispatcher's `from_phone`.

Job # format: last 6 chars of UUID, uppercased — consistent with conventions.md and E02S02.

### AC4 — If no tech assigned, tech name is omitted gracefully

**Status: Implemented**

The `Build Guard SMS` Code node checks:
```javascript
const techName = job.techs && job.techs.name ? job.techs.name : null;
if (techName) {
  guardMsg = `Job #${jobShort} is already ${currentStatus}. Assigned to ${techName}.`;
} else {
  guardMsg = `Job #${jobShort} is already ${currentStatus}.`;
}
```

When `tech_id` is NULL, Supabase returns `job.techs = null`. The tech name clause is omitted from the message.

### AC5 — SMS sent from the same Twilio number as the original job alert

**Status: Implemented**

`Build Guard SMS` sets `from_number = $('Get Client by Phone').item.json.twilio_number`.
`Send Guard SMS` native Twilio node uses `from_number` as the `from` field.
This is the same `clients.twilio_number` field used by all other SMS nodes in this workflow
(`Send Confirmation SMS`, `Send Error SMS`). Never hardcoded — always pulled from Supabase.

## Node Details

### Get Current Job State (HTTP Request, typeVersion 4.2)

**URL:** `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs?client_id=eq.[id]&select=id,status,tech_id,techs(name)&order=created_at.desc&limit=1`

The `?select=techs(name)` syntax uses Supabase's foreign key join via `jobs_tech_id_fkey`.
Returns `techs: { name: "Mike" }` when assigned, `techs: null` when not assigned.
`order=created_at.desc&limit=1` — most recent job for this client, any status.

Headers: `apikey`, `Authorization: Bearer`, `Content-Type: application/json` — `SUPABASE_ANON_KEY_PLACEHOLDER` in both key headers.

### Build Guard SMS (Code, typeVersion 2)

Reads from `Get Current Job State` via `$input.item.json` (direct — this node is immediately downstream).
Falls back gracefully if no job row is found (edge case where client has no jobs at all).
Output: `{ guard_msg, dispatcher_phone, from_number }`.

### Send Guard SMS (Twilio, typeVersion 1)

Native n8n Twilio node. `continueOnFail: true` — SMS failure does not block workflow.
Credential: `twilio-credentials` (same as all other Twilio nodes in this workflow).


## Rules Applied

- HTTP Request node for Supabase GET — no native Supabase node (conventions.md)
- Native n8n Twilio node for SMS — not HTTP Request (conventions.md, DEC-006)
- n8n v2 expression pattern: `$('Node Name').item.json.field` in Code node for upstream nodes
- `$input.item.json` used in `Build Guard SMS` for its direct upstream node
- Code node output: `return [{ json: { ... } }]`
- `SUPABASE_ANON_KEY_PLACEHOLDER` used in `Get Current Job State` HTTP Request node
- `continueOnFail: true` on Twilio node — informational SMS, not a blocking step
- Job # format: `.slice(-6).toUpperCase()` per conventions.md

## Known Risks / Limitations

1. `SUPABASE_ANON_KEY_PLACEHOLDER` in `Get Current Job State` — human must replace before activation.
2. n8n v2 IF false-branch wiring bug — human must verify `Job is New?` FALSE → `Get Current Job State` after import (same as `Tech Found?` → `Fetch All Techs`).
3. If client has no jobs at all: `Get Current Job State` returns empty array → `job.id` is undefined → fallback message sent (`No active job found for your number.`). This is an edge case; normal operation always has at least one job.
