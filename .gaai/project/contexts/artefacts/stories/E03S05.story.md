# Story E03S05: Tech Dashboard — En Route and On Site Action Buttons

## User Story

**As a** technician,
**I want** large, easy-to-tap buttons to update my job status to En Route or On Site,
**so that** I can notify the customer with one tap while driving or walking up to the job.

---

## Context

The action buttons are the core interaction on the tech dashboard. They must be large enough to tap reliably on a phone screen with one hand. Each button triggers a status update in Supabase and a downstream n8n workflow (E03S06, E03S07) that sends the customer SMS. Button state must reflect current job status to prevent double-taps.

---

## Acceptance Criteria

- [ ] **AC1:** Each job card displays an **En Route** button when job status is `Assigned`.
- [ ] **AC2:** Each job card displays an **On Site** button when job status is `En Route`.
- [ ] **AC3:** When status is `On Site`, no further action buttons are shown on the card (awaiting Job Complete in Sprint 4).
- [ ] **AC4:** Tapping En Route sends a PATCH to `jobs` table: `{ status: 'En Route' }` for that job ID.
- [ ] **AC5:** Tapping On Site sends a PATCH to `jobs` table: `{ status: 'On Site' }` for that job ID.
- [ ] **AC6:** Button tap target is minimum 48×48px — large enough for reliable one-handed use on mobile.
- [ ] **AC7:** After a successful PATCH, the button is replaced with the next-state button (or removed if On Site) without a full page reload.
- [ ] **AC8:** If the PATCH fails, an inline error message is shown on the card — button is re-enabled so the tech can retry.
- [ ] **AC9:** During the PATCH request, the button is disabled to prevent double-submission.

---

## Out of Scope

- Job Complete button (Sprint 4)
- Customer SMS notification (triggered by n8n via Supabase real-time or webhook — E03S06, E03S07)
- Reassignment or job rejection

---

## Dependencies

- E03S04 — buttons live inside the tech dashboard job cards

---

## Notes

- PATCH endpoint: `PATCH /rest/v1/jobs?id=eq.[jobId]` with `{ status: 'En Route' }` or `{ status: 'On Site' }`
- Auth header: use anon key — `techs` token does not issue a Supabase JWT; RLS anon UPDATE policy on `jobs` already permits this
- Status transition must also update dispatcher dashboard via real-time (E02S09 already handles incoming UPDATE events — no additional work needed)
- Visual design: full-width button at card bottom, high-contrast color (En Route = amber, On Site = green to match dispatcher badge colors)
- Test on actual mobile device before marking done — not just browser dev tools
