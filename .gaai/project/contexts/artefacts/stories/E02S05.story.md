# Story E02S05: Dispatcher Dashboard — Job List View

## User Story

**As a** dispatcher,
**I want** to see a live list of all current jobs with key details on a single dashboard page,
**so that** I can quickly understand what's in the queue, what's urgent, and who has been assigned what.

---

## Context

The dispatcher needs a central view of all jobs before they can make assignment decisions — either by SMS reply (E02S01–S04) or via the dashboard dropdown (E02S07). This dashboard is the primary tool for the dispatcher's daily workflow. It is a single `index.html` deployed to Netlify, following the AC demo canonical template pattern.

---

## Acceptance Criteria

- [ ] **AC1:** When the dashboard loads, a job list is displayed showing the following columns for every job: Job ID, Customer Name, Address, Service Type, Urgency, Status, Assigned Tech, Submitted Time.
- [ ] **AC2:** Jobs are ordered by Submitted Time — oldest (most urgent by wait time) first.
- [ ] **AC3:** If a job has no assigned tech, the Assigned Tech column displays "Unassigned".
- [ ] **AC4:** All jobs currently in the `jobs` table are shown — no filtering or pagination required at this stage.
- [ ] **AC5:** If the jobs table is empty, the dashboard shows a clear empty-state message (e.g., "No jobs in queue").
- [ ] **AC6:** The dashboard is usable on both desktop and mobile screens — columns readable without horizontal scroll on a standard mobile viewport (375px width).
- [ ] **AC7:** Job data is fetched from the database on page load without requiring manual refresh.
- [ ] **AC8:** If the data fetch fails, an error message is displayed to the dispatcher rather than a blank or broken UI.

---

## Out of Scope

- Authentication / login (covered by E02S08)
- Urgency visual flagging / color badges (covered by E02S06)
- Assignment dropdown interaction (covered by E02S07)
- Real-time auto-update on status change (covered by E02S09)
- Pagination or filtering
- Multiple client views — this dashboard is for a single client's jobs

---

## Dependencies

## Notes

- Follow AC demo (Cool Breeze HVAC) as the canonical `index.html` template — all CSS and JS inline, no separate files.
- Data source: Supabase `jobs` table. Use HTTP Request node pattern — not native Supabase integration (DEC-003).
- Supabase returns arrays — field access pattern: `response[0].field_name` in JS fetch.
- Required fields from `jobs` table: `id`, `customer_name`, `address`, `service_type`, `is_urgent`, `status`, `tech_name` (or join to `techs` table), `created_at`.
- No auth on this story — auth layer is added by E02S08 which depends on this.
- Deployed as subdomain under `sendjob.app` via Netlify CLI (`netlify deploy --prod`).
- Risk: `jobs` table could grow large over time — no pagination required now, but note for future backlog.
