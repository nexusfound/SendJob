# Story E02S04: n8n — "Already Assigned" SMS Guard

## User Story

**As a** dispatcher,
**I want** to be notified if I try to assign a job that's no longer available for assignment,
**so that** I don't accidentally trigger a duplicate assignment or cause confusion for the tech.

---

## Context

The dispatcher SMS reply window is status-based: a reply is only valid while the job is in `New` status. If the job has already been assigned (via dashboard or another SMS), the system must reject the assignment attempt and inform the dispatcher rather than silently failing or double-assigning.

---

## Acceptance Criteria

- [ ] **AC1:** Before attempting to assign a tech, n8n checks `jobs.status` for the relevant job.
- [ ] **AC2:** If `jobs.status` is not `New`, no assignment is made — the job record is not modified.
- [ ] **AC3:** The dispatcher receives an SMS: `"Job #[ID] is already [status]. Assigned to [Tech Name]."` — where status and tech name reflect the current job state.
- [ ] **AC4:** If the job has no assigned tech (e.g., status is something other than New/Assigned), the message omits the tech name gracefully.
- [ ] **AC5:** The SMS is sent from the same Twilio number as the original job alert.

---

## Out of Scope

- Allowing the dispatcher to override an existing assignment via SMS
- Handling any status other than New as an edge case (En Route, On Site, Complete all treated the same — "already assigned")

---

## Dependencies

- E02S02 — the status check is a gate within the E02S02 flow, before the PATCH is executed.

---

## Notes

- This is a pre-assignment status gate in the n8n workflow — check `jobs.status` immediately after identifying the job, before any PATCH.
- Fetch current job state: `GET /rest/v1/jobs?id=eq.{job_id}&select=status,tech_id,techs(name)` — use join to get tech name in one call.
- Use native n8n Twilio node for the guard SMS (NOT HTTP Request).
- Decision rule: `if status == 'New' → proceed to E02S02 assignment logic; else → send guard SMS`.
