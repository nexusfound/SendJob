# Story E04S03: E2E Test Script — Full Job Lifecycle

## User Story

**As a** founder preparing to onboard the first pilot client,
**I want** a documented manual test checklist covering the full job lifecycle,
**so that** I can validate the entire system with real SMS and real data before going live.

---

## Context

Sprint 4 closes the feature loop. Before Sprint 5 (Pilot Client), a single full E2E run confirms every integration works together in production. The test uses a real test job, real phone numbers, and real SMS delivery — not mocks or stubs.

---

## Acceptance Criteria

- [ ] **AC1:** Test script file exists at `.gaai/project/contexts/artefacts/test-scripts/e2e-sprint4.md`.
- [ ] **AC2:** Script covers all 13 steps of the job lifecycle as defined in `project/context.md`.
- [ ] **AC3:** Each step includes: action to perform, expected result, and a pass/fail checkbox.
- [ ] **AC4:** Script includes a prerequisites section listing required test data: client record, tech record, test phone numbers for customer and dispatcher.
- [ ] **AC5:** Script flags known risk: Twilio A2P 10DLC — if approval is still pending, outbound SMS to non-verified numbers may fail. Note workaround (use Twilio-registered test number as recipient).
- [ ] **AC6:** Script has been run once in full — all 13 steps pass before this story is marked done.
- [ ] **AC7:** Any failures found during the test run are added to the backlog as new stories before this story is marked done.

---

## Out of Scope

- Automated test tooling
- Load or performance testing
- Testing error paths (expired token, invalid tech name, etc.) — covered by individual story ACs

---

## Dependencies

- E04S01, E04S02 — all Sprint 4 features complete before running the test
- All Sprint 2 (E02) and Sprint 3 (E03) stories — E2E covers the full stack

---

## Notes

- Test client record must be separate from any pilot client's production record
- Run the dispatcher dashboard on desktop, tech dashboard on mobile simultaneously
- Document any timing observations (e.g., SMS delivery lag) in the script — useful for setting pilot client expectations
- The test script is a living document — update it as new sprints add steps to the lifecycle
