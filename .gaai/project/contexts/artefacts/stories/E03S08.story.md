# Story E03S08: Tech Dashboard — Expired / Invalid Token Error Page

## User Story

**As a** technician with an expired or invalid magic link,
**I want** to see a clear, friendly message explaining what happened and what to do next,
**so that** I'm not left with a broken blank page and no idea how to get back into my jobs.

---

## Context

Magic link tokens expire or get replaced when a new job is assigned. A tech might tap an old link from their SMS history. Rather than a blank page or a technical error, they should see a human-friendly message that tells them exactly what to do.

---

## Acceptance Criteria

- [ ] **AC1:** If `?token=` is missing from the URL, the error state is shown immediately — no Supabase query attempted.
- [ ] **AC2:** If the token does not match any row in `techs.magic_link_token`, the error state is shown.
- [ ] **AC3:** Error message displayed: "This link has expired. Text JOBS to [twilio_number] to get a new one." where `[twilio_number]` is the client's Twilio number.
- [ ] **AC4:** The error page is styled consistently with the tech dashboard (same dark navy design, not a raw browser error).
- [ ] **AC5:** The error state is shown in place of the job list — no partial dashboard content is visible.
- [ ] **AC6:** If the token query fails due to a network error (not a missing token), a separate generic error message is shown: "Something went wrong. Please try again or contact your dispatcher."

---

## Out of Scope

- Automatic token refresh
- "Request new link" button (requires backend endpoint not in scope)
- Token expiry based on time (Sprint 4)

---

## Dependencies

- E03S04 — error state is part of the same `index.html` as the tech dashboard

---

## Notes

- Token lookup: `GET /rest/v1/techs?magic_link_token=eq.[token]&select=id,name,client_id`
- If result is empty array → show expired/invalid error (AC2, AC3)
- Client's Twilio number: once token is invalid, client_id is unknown — use a fallback number or generic instruction ("contact your dispatcher")
- Keep the error page simple — techs read this on a phone, possibly frustrated. One clear action only.
- The "Text JOBS to [number]" prompt is a future n8n flow (not in Sprint 3 scope) — the message sets the expectation without requiring the feature to exist yet
