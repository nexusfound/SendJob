# Story E02S07: Dashboard — Dropdown Assignment

## User Story

**As a** dispatcher,
**I want** to assign a technician to a job directly from the dashboard using a dropdown,
**so that** I can assign jobs without sending an SMS.

---

## Context

The SMS path (E02S01–S04) is one assignment method. The dashboard provides a second path. When the dispatcher selects a tech from a dropdown on a job row, the same state change occurs in Supabase — the two paths must produce identical results.

---

## Acceptance Criteria

- [ ] **AC1:** Each job row in `New` status shows a tech dropdown (or "Assign" button + dropdown) listing all active techs for that client, loaded from the `techs` table filtered by `client_id`.
- [ ] **AC2:** Selecting a tech and confirming sends a PATCH to the `jobs` table: `tech_id` = selected tech's UUID, `status` = `Assigned`, `assigned_at` = current timestamp — identical to the SMS assignment path.
- [ ] **AC3:** After a successful assignment, the row visually updates to show the assigned tech name and `Assigned` status — without a full page reload.
- [ ] **AC4:** Jobs that are already `Assigned`, `En Route`, `On Site`, or `Complete` do not show an active dropdown — they display the assigned tech name as read-only.
- [ ] **AC5:** The dropdown is usable on mobile (375px) — meets the 52px minimum tap target from the design system.
- [ ] **AC6:** If the PATCH request fails, an error is shown to the dispatcher — the row does not silently show the wrong state.
- [ ] **AC7:** The Supabase PATCH uses the browser JS fetch pattern with `apikey` and `Authorization: Bearer` headers (same as E02S05 data fetch).

---

## Out of Scope

- Triggering the n8n magic link / tech SMS notification (Sprint 3 — tech flow)
- Unassigning or reassigning a job via the dashboard
- Bulk assignment

---

## Dependencies

- E02S05 — dashboard and job table must exist.

---

## Notes

- Dropdown populated from: `GET /rest/v1/techs?client_id=eq.{client_id}&select=id,name` — need to know the client_id. For now, hardcode the test client UUID or read it from `clients` table by subdomain.
- PATCH endpoint: `PATCH /rest/v1/jobs?id=eq.{job_id}` with body `{ "tech_id": "...", "status": "Assigned", "assigned_at": new Date().toISOString() }`.
- Add `Prefer: return=representation` and `Content-Type: application/json` headers on PATCH.
- Design system: dropdown must use `--navy-mid` background, `--border` border, `--text` color, `16px` font size (prevents iOS zoom), `min-height: 52px`.
- The `techs` join pattern (`?select=*,techs(name)`) used in E02S05 fetch can also be used here to get tech names without a second query.
