# Story E04S02: N8N — Job Complete → Customer SMS with Payment Link

## User Story

**As a** customer whose job has just been completed,
**I want** to receive an SMS with a direct payment link immediately after the technician marks the job done,
**so that** I can pay without needing to call or wait for an invoice.

---

## Context

Mirror of E03S06 (En Route SMS) and E03S07 (On Site SMS). Uses the same polling + sentinel pattern established in DEC-016. Requires one new sentinel column on the `jobs` table. Includes an additional guard for null `payment_link` — not needed in prior notification stories.

---

## Acceptance Criteria

- [ ] **AC1:** `job_complete_sms_sent boolean DEFAULT false` column added to `jobs` table via Supabase SQL migration before workflow is activated.
- [ ] **AC2:** N8N workflow uses Schedule Trigger polling every 30 seconds.
- [ ] **AC3:** Poll filter: `status=eq.Complete&job_complete_sms_sent=eq.false` — returns only unprocessed completed jobs.
- [ ] **AC4:** Workflow fetches `clients.business_name`, `clients.payment_link`, and `clients.twilio_number` using `jobs.client_id`.
- [ ] **AC5:** IF `payment_link` is null or empty string → skip SMS node, set sentinel to true, no error raised.
- [ ] **AC6:** IF `customer_phone` is null or empty → skip SMS node, set sentinel to true, no error raised.
- [ ] **AC7:** SMS sent via native n8n Twilio node. `From` = `clients.twilio_number`. `To` = `jobs.customer_phone`.
- [ ] **AC8:** SMS text: `[Business Name]: Your job is complete. Pay here: [payment_link]` (exact template from conventions.md).
- [ ] **AC9:** After SMS send (or skip), PATCH `jobs`: `job_complete_sms_sent = true`.
- [ ] **AC10:** Both Twilio node and PATCH node have `continueOnFail: true` — SMS failure never corrupts job status.
- [ ] **AC11:** If poll returns multiple jobs, Split In Batches node processes them one at a time.

## Notes

### Schema migration (run before activating workflow)
```sql
ALTER TABLE jobs ADD COLUMN job_complete_sms_sent boolean DEFAULT false;
```

### N8N workflow structure (follow DEC-016 exactly)
```
Schedule Trigger (30s)
  → HTTP Request: GET /rest/v1/jobs?status=eq.Complete&job_complete_sms_sent=eq.false&select=*
  → IF: items exist (length > 0)
    TRUE → Split In Batches
      LOOP → HTTP Request: GET /rest/v1/clients?id=eq.[client_id]&select=business_name,payment_link,twilio_number
           → IF: customer_phone not null AND payment_link not null
               TRUE  → Twilio node (native): send SMS
                     → HTTP Request: PATCH /rest/v1/jobs?id=eq.[job_id] { job_complete_sms_sent: true }
               FALSE → HTTP Request: PATCH /rest/v1/jobs?id=eq.[job_id] { job_complete_sms_sent: true }
    FALSE → (no-op)
```

- **Import bug:** After JSON import, rewire Split In Items to **loop** output — it always wires to done on import (known n8n v2.6.4 bug)
- Supabase returns array — access client fields as `$('Node Name').item.json[0].business_name`
- Use `$('Node Name').item.json.field` pattern for all upstream node references — never `$json.field`
- Both Twilio node and PATCH node: enable `continueOnFail` in node settings
