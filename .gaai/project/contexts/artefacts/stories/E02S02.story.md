# Story E02S02: n8n — Parse SMS Reply, Match Tech, Mark Job Assigned

## User Story

**As a** dispatcher,
**I want** my SMS reply with a tech name to automatically assign that tech to the job and update its status,
**so that** the job moves from New to Assigned without any manual dashboard interaction.

---

## Context

Once the inbound SMS is received (E02S01), this story handles the logic: extract the tech name from the message, look it up against the client's tech roster in Supabase, and update the job record. This is the core of the SMS assignment path.

---

## Acceptance Criteria

- [ ] **AC1:** n8n extracts the tech name or code from the incoming SMS body (case-insensitive match).
- [ ] **AC2:** n8n queries the `techs` table filtered by `client_id` to find a matching tech — match is scoped to that client only (no cross-client matching).
- [ ] **AC3:** On a successful match, n8n sends a PATCH request to the `jobs` table setting: `tech_id` = matched tech's UUID, `status` = `Assigned`, `assigned_at` = current timestamp.
- [ ] **AC4:** After successful assignment, the dispatcher receives a confirmation SMS: `"Job #[ID] assigned to [Tech Name]."` (Job # = last 6 chars of UUID, uppercased).
- [ ] **AC5:** If the job status is not `New` at the time of the SMS reply, the assignment does not proceed — this case is handled by E02S04.
- [ ] **AC6:** All Supabase queries use HTTP Request nodes — not native Supabase nodes (bugged in n8n v2.6.4).

---

## Out of Scope

- Error SMS for unrecognized tech name (E02S03)
- Already-assigned guard SMS (E02S04)
- Magic link generation and tech notification (Sprint 3)

---

## Dependencies

- E02S01 — inbound Twilio webhook must be wired and receiving messages.

## Notes

- HTTP Request to Supabase `techs` table: `GET /rest/v1/techs?client_id=eq.{client_id}&name=ilike.{reply_text}` — use `ilike` for case-insensitive match.
- Supabase returns arrays in n8n 2.x HTTP Request node — access matched tech as `$('Node Name').item.json.id` (first element returned directly, not `$json[0].id`).
- PATCH to `jobs` table: `PATCH /rest/v1/jobs?id=eq.{job_id}` with body `{ "tech_id": "...", "status": "Assigned", "assigned_at": "now()" }`.
- Add `Prefer: return=representation` header on PATCH if the updated row is needed downstream.
- The inbound SMS `From` field identifies the dispatcher; use it to look up the client row in `clients` table by `dispatcher_phone` to get `client_id`.
- Use native n8n Twilio node for sending the confirmation SMS (NOT HTTP Request).
