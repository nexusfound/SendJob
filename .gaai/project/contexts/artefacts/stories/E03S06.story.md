# Story E03S06: N8N — En Route Action: Customer SMS + Status Update

## User Story

**As a** customer,
**I want** to receive an SMS when the technician is on the way,
**so that** I know when to expect them and can be ready.

---

## Context

When the tech taps En Route (E03S05), the Supabase `jobs` table is updated to `En Route`. N8N listens for this change (via webhook or polling) and sends the customer an SMS. The message uses the canonical En Route template from conventions.

---

## Acceptance Criteria

- [ ] **AC1:** When `jobs.status` changes to `En Route`, n8n sends an SMS to `jobs.customer_phone` within 60 seconds.
- [ ] **AC2:** SMS text: `[Business Name]: Your technician is on the way.`
- [ ] **AC3:** Business name is pulled from `clients.business_name` via `jobs.client_id` — never hardcoded.
- [ ] **AC4:** Uses n8n native Twilio node — NOT HTTP Request (per conventions).
- [ ] **AC5:** If the customer has no phone (`customer_phone` is null), the SMS step is skipped — workflow does not error.
- [ ] **AC6:** Job status in Supabase remains `En Route` regardless of SMS delivery outcome — SMS failure must not revert the status.

---

## Out of Scope

- Email fallback for customer notification
- Estimated arrival time
- Tech location tracking

---

## Dependencies

- E03S05 — status change to `En Route` is the trigger

---

## Notes

- N8N trigger options: (1) Supabase webhook via pg_net → n8n webhook, or (2) n8n polling every 30s on `jobs` where `status = 'En Route' AND en_route_sms_sent IS NULL`. Polling is simpler to implement. Webhook is more reliable at scale. Polling acceptable for Sprint 3.
- If using polling: add `en_route_sms_sent` boolean column to `jobs` table to prevent duplicate sends (default false, set true after send)
- Twilio node: use `clients.twilio_number` as the From number — always per-client, never shared
- Business name lookup: `GET /rest/v1/clients?id=eq.[client_id]&select=business_name`
- n8n HTTP Request to Supabase — use apikey + Authorization Bearer headers
