## High-Level Plan

### Objective

When a job is assigned (status changes to `Assigned`), n8n must:
1. Generate a cryptographically random UUID token
2. PATCH `techs.magic_link_token` and `techs.token_issued_at` for the assigned tech
3. Handle PATCH failure gracefully — log error, do NOT roll back the assignment

This workflow is triggered as a **sub-flow** by both E02S02 (SMS assignment path) and E02S07 (dashboard assignment path), or implemented as a shared node sequence appended to each of those existing workflows.

### Trigger Mechanism (AC4)

Two workflows currently handle job assignment:
- **E02S02 workflow** (SMS reply path): sets `jobs.status = Assigned`, sets `jobs.tech_id`
- **E02S07 dashboard path**: sets `jobs.status = Assigned`, sets `jobs.tech_id`

The token generation must execute after `tech_id` is written to the jobs table in both paths. The implementation approach:
- **Preferred:** Add token-generation node sequence (Code node + HTTP PATCH + error branch) at the end of both the E02S02 n8n workflow and the E02S07 dashboard-triggered n8n workflow (or a shared sub-workflow called by both).
- **Alternative:** n8n polling trigger watching for `jobs.status = Assigned` (not preferred — adds latency and polling overhead).

**Decision:** Append the token-generation node sequence to both existing workflows. This is minimal, deterministic, and consistent with the existing pattern. No polling.

## Node Sequence Design

### Node 1 — Generate Token (Code node)

**Node type:** Code
**Name:** `Generate Magic Link Token`
**Code:**
```javascript
return [{ json: { token: require('crypto').randomUUID() } }];
```
**Output:** `{ token: "uuid-string" }`
**Input dependency:** Upstream node that resolved `tech_id` (from job assignment step)

### Node 2 — PATCH techs table (HTTP Request node)

**Node type:** HTTP Request
**Name:** `PATCH techs — Magic Link Token`
**Method:** PATCH
**URL:** `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/techs?id=eq.{{ $('Assign Job').item.json.tech_id }}`

> Note: The exact upstream node name referencing `tech_id` will differ between E02S02 and E02S07 workflows. The implementation must use the correct `$('Node Name').item.json.tech_id` reference for each workflow.

**Headers:**
```
apikey: [SUPABASE_ANON_KEY]
Authorization: Bearer [SUPABASE_ANON_KEY]
Content-Type: application/json
Prefer: return=representation
```

**Body (JSON):**
```json
{
  "magic_link_token": "{{ $('Generate Magic Link Token').item.json.token }}",
  "token_issued_at": "{{ $now.toISO() }}"
}
```

**Continue on Fail:** TRUE — this is the critical setting that prevents PATCH failure from halting the workflow. (AC5)

### Node 3 — Error Log Branch (IF node + Set node)

**Purpose:** After the PATCH node, check if the HTTP Request returned an error status or empty response. If the PATCH failed, log the error. The job assignment is already written to the `jobs` table at this point — no rollback.

**Node type:** IF
**Name:** `Token PATCH Succeeded?`
**Condition:** `{{ $('PATCH techs — Magic Link Token').item.json.id }}` is not empty (representation response returns the patched row)

- **True branch:** workflow continues (or ends — this is the last step before E03S03 SMS trigger)
- **False branch:** Set node `Log Token Error` — outputs `{ error: "magic_link_token PATCH failed", tech_id: "...", timestamp: "..." }` — workflow ends here on false branch without reverting assignment

## Edge Cases

### AC1 — Token generation on job assignment

- Happy path: Code node executes `crypto.randomUUID()`, returns a valid UUID-format string
- Boundary: n8n Code node always has `crypto` available in the Node.js runtime — no import error
- Error path: Code node throws → n8n marks execution error — this is an upstream node failure, separate from AC5 PATCH failure handling

### AC2 — Token stored in techs.magic_link_token and techs.token_issued_at

- Happy path: PATCH returns HTTP 200 with `return=representation` — patched row confirms both fields written
- Boundary: `tech_id` is a valid UUID FK in the `techs` table — filter `id=eq.[tech_id]` returns exactly one row
- Error path: PATCH fails (network error, RLS policy rejects) → `Continue on Fail` = true → error branch logs, workflow does not crash

### AC3 — New assignment overwrites previous token

- Happy path: PATCH with `?id=eq.[tech_id]` always overwrites — no INSERT, no duplicate rows
- Boundary: Tech with existing token gets a new job → previous token is overwritten as designed
- Error path: Same as AC2 error path

### AC4 — Both assignment paths trigger token generation

- Happy path: E02S02 workflow has token-generation nodes appended after `jobs.status = Assigned` write; E02S07 workflow has the same sequence appended
- Boundary: Both paths use the same PATCH URL pattern and node logic — the only difference is the upstream node name for `tech_id`
- Error path: If one workflow is missing the token nodes, AC4 fails — implementation must verify both workflows

### AC5 — PATCH failure does not roll back assignment

- Happy path: PATCH succeeds — token stored, workflow continues
- Critical path: PATCH fails → `Continue on Fail: true` on HTTP Request node → IF node routes to error log branch → workflow ends cleanly. The `jobs.status = Assigned` and `jobs.tech_id` written in the prior step are NOT affected.
- This is the most important rule to enforce — it must be verified in QA
