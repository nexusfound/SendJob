# Execution Plan — E02S09: Dashboard Real-Time Job Status Updates

## Summary

Add Supabase real-time subscription to `dispatcher-dashboard/index.html` so the job table
updates automatically on INSERT/UPDATE without polling. Must use the existing `supabaseClient`
instance (from E02S08 auth) and respect the active auth session.

---

## Files Affected

| File | Change |
|---|---|
| `dispatcher-dashboard/index.html` | Add real-time subscription logic, update status indicator states, add disconnect handling |

No other files touched.

---

## Approach

### 1. Status Indicator — new `live` state

The existing status indicator has: `loading` (pulsing blue) → `loaded` (green) → `error` (red).

Add a `live` state: pulsing green dot, label "Live". This is set after the initial `loadJobs()`
call succeeds AND the real-time subscription confirms it is active (`.subscribe()` callback
with status `SUBSCRIBED`).

CSS addition:
```css
.status-dot.live {
  background: var(--success);
  animation: pulse 1.2s ease-in-out infinite;
}
```

### 2. Real-time subscription setup

Use the existing `supabaseClient` — do NOT create a new client (AC7, story notes).

Pattern from story notes:
```javascript
supabase.channel('jobs')
  .on('postgres_changes', { event: '*', schema: 'public', table: 'jobs' }, handleChange)
  .subscribe();
```

Subscribe AFTER `loadJobs()` resolves (to avoid race condition between initial render and
first real-time event). Subscribe inside `showAppView()` after `loadJobs()` call.

Channel held in a module-level variable `realtimeChannel` so it can be unsubscribed on logout.

### 3. handleChange — dispatch on eventType

```javascript
function handleRealtimeChange(payload) {
  var eventType = payload.eventType; // 'INSERT' | 'UPDATE' | 'DELETE'
  if (eventType === 'INSERT') {
    handleInsert(payload.new);
  } else if (eventType === 'UPDATE') {
    handleUpdate(payload.new);
  }
  // DELETE: out of scope per story
}
```

**INSERT:** Append new row to `tbodyEl` (oldest-first maintained because new rows go to bottom).
If the table is in empty/error state, transition to table state first.

**UPDATE:** Find existing `<tr>` by job ID (`tr[data-job-id]` attribute added to each row),
then update the Status, Assigned Tech, and Urgency cells in-place.

To support row lookup by job ID, add `data-job-id` attribute to each `<tr>` in `buildJobRow()`.

### 4. Subscription lifecycle

- On `subscribe()` callback with status `SUBSCRIBED` → call `setLiveState()`
- On `subscribe()` callback with status `CHANNEL_ERROR` or `TIMED_OUT` → call `setDisconnectedState()`
- On logout: call `supabaseClient.removeChannel(realtimeChannel)` then set channel to null

### 5. Reconnection / graceful degradation (AC6)

Supabase real-time JS client handles reconnection internally (exponential backoff).
On error: show disconnected state in the indicator. Last known data stays in the table.
Do NOT crash. Do NOT clear the table on disconnect.

### 6. Acceptance criteria mapping

| AC | Addressed by |
|---|---|
| AC1 | UPDATE handler updates row in-place within 2s (Supabase real-time typically <500ms) |
| AC2 | INSERT handler appends row; oldest-first order maintained (new rows go to bottom) |
| AC3 | No polling, no `setInterval` — subscription only |
| AC4 | Documentation note in code comment; pre-condition for human to action |
| AC5 | `live` CSS state + `setLiveState()` function |
| AC6 | Error handler sets disconnected state; table data preserved |
| AC7 | Uses existing `supabaseClient` — same auth session from E02S08 |

---

## Edge Cases

- Real-time event arrives before `techsList` is loaded: INSERT/UPDATE will show "Unassigned"
  in the tech column. Acceptable — tech name will appear after next assignment action.
- Real-time UPDATE for a job not in the current DOM (possible if table was in empty state):
  treat as INSERT and append.
- If dashboard is on login view when real-time event arrives: no-op (channel not subscribed
  until authenticated).

Low. Single file. Additive only — no existing functionality removed or altered. Auth session
integration uses read-only access to `supabaseClient.auth.getSession()` (no changes to auth flow).
