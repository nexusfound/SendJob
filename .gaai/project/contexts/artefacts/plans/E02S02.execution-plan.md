# Execution Plan — E02S02: Parse SMS Reply, Match Tech, Mark Job Assigned

## Summary

This workflow extends the E02S01 `sendjob-intake` n8n workflow. After `Parse SMS Fields`
extracts `message_body`, `from_phone`, and `to_phone`, this story adds the following logic:

1. Look up client record by `from_phone` (dispatcher phone) → get `client_id`
2. Extract tech name from `message_body` (case-insensitive, trimmed)
3. Query `techs` table by `client_id` + `name ilike` → get matched tech (or no match)
4. Query `jobs` table to find the latest `New` job for this client → get `job_id`
5. Guard: if job status is not `New` → exit branch (E02S04 handles this case)
6. PATCH `jobs` table: set `tech_id`, `status = Assigned`, `assigned_at = now()`
7. Send confirmation SMS to dispatcher via native Twilio node

All Supabase interactions use HTTP Request nodes only. Twilio uses native n8n Twilio node.

---

## Approach Evaluation

**Skipped.** All approaches are established conventions in `conventions.md`:
- Supabase queries: HTTP Request nodes (GET with query params, PATCH with body)
- Twilio SMS: native n8n Twilio node
- Node expressions: `$('Node Name').item.json.field`
- n8n v2 array access: HTTP Request returns first element directly

## Implementation Sequence

### Phase 1 — Client Lookup by Dispatcher Phone (AC2, AC5)

**Step 1.1 — GET clients table by dispatcher_phone**

Add an HTTP Request node immediately after `Parse SMS Fields`:

- Node name: `Get Client by Phone`
- Type: HTTP Request (typeVersion 4.2)
- Method: GET
- URL: `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/clients`
- Query parameters:
  - `dispatcher_phone`: `eq.{{ $('Parse SMS Fields').item.json.from_phone }}`
  - `select`: `id,business_name,twilio_number`
- Headers:
  - `apikey`: `[SUPABASE_ANON_KEY]`
  - `Authorization`: `Bearer [SUPABASE_ANON_KEY]`

Expected result: array with one client record. Access as:
- `$('Get Client by Phone').item.json.id` → client UUID
- `$('Get Client by Phone').item.json.business_name` → e.g. "AC Express"
- `$('Get Client by Phone').item.json.twilio_number` → e.g. "+13212489804"

**Step 1.2 — IF node: client found guard**

Add an IF node after `Get Client by Phone`:

- Condition: `$('Get Client by Phone').item.json.id` is not empty
- True branch → continue to tech lookup
- False branch → no-op (drop silently — unknown sender, out of scope)

---

### Phase 2 — Extract Tech Name from SMS Body (AC1)

**Step 2.1 — Code node: Extract Tech Name**

Add a Code node after the IF true branch:

- Node name: `Extract Tech Name`
- Type: Code (typeVersion 2)
- Code:
```javascript
const rawText = $('Parse SMS Fields').item.json.message_body || '';
const techName = rawText.trim();
return [{ json: { tech_name: techName } }];
```

Output field: `tech_name` — raw trimmed text from dispatcher SMS.

Note: Full case-insensitive matching is handled in Phase 3 via Supabase `ilike` operator.
No need to lowercase here — ilike handles it server-side.

---

### Phase 3 — Tech Lookup in Supabase (AC1, AC2)

**Step 3.1 — GET techs table by client_id + name ilike**

Add an HTTP Request node after `Extract Tech Name`:

- Node name: `Get Tech by Name`
- Type: HTTP Request (typeVersion 4.2)
- Method: GET
- URL: `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/techs`
- Query parameters:
  - `client_id`: `eq.{{ $('Get Client by Phone').item.json.id }}`
  - `name`: `ilike.{{ $('Extract Tech Name').item.json.tech_name }}`
  - `select`: `id,name,phone`
- Headers:
  - `apikey`: `[SUPABASE_ANON_KEY]`
  - `Authorization`: `Bearer [SUPABASE_ANON_KEY]`

Expected result: array with matching tech, or empty array if no match.
Access matched tech as: `$('Get Tech by Name').item.json.id` (first element, n8n v2 direct access)

**Step 3.2 — IF node: tech found guard**

Add an IF node after `Get Tech by Name`:

- Condition: `$('Get Tech by Name').item.json.id` is not empty
- True branch → continue to job lookup
- False branch → exit (E02S03 handles "no match" SMS error — out of scope here)

---

### Phase 4 — Job Lookup and Status Guard (AC3, AC5)

**Step 4.1 — GET latest New job for this client**

Add an HTTP Request node after tech found IF true branch:

- Node name: `Get Latest New Job`
- Type: HTTP Request (typeVersion 4.2)
- Method: GET
- URL: `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs`
- Query parameters:
  - `client_id`: `eq.{{ $('Get Client by Phone').item.json.id }}`
  - `status`: `eq.New`
  - `select`: `id,status,customer_name`
  - `order`: `created_at.desc`
  - `limit`: `1`
- Headers:
  - `apikey`: `[SUPABASE_ANON_KEY]`
  - `Authorization`: `Bearer [SUPABASE_ANON_KEY]`

Expected result: one job record (most recent New job), or empty if none.
Access as: `$('Get Latest New Job').item.json.id` → job UUID

**Step 4.2 — IF node: job is New guard (AC5)**

Add an IF node after `Get Latest New Job`:

- Condition: `$('Get Latest New Job').item.json.status` equals `New`
- True branch → proceed to assignment PATCH
- False branch → exit (E02S04 handles already-assigned guard — out of scope here)

Note: This guard also handles the case where no New job exists — if no job returned,
`status` field will be undefined/empty, which fails the `equals New` check. Both cases
(wrong status and no job) exit cleanly.

---

### Phase 5 — Assign Job (AC3)

**Step 5.1 — PATCH jobs table**

Add an HTTP Request node after the job-is-New IF true branch:

- Node name: `Assign Job`
- Type: HTTP Request (typeVersion 4.2)
- Method: PATCH
- URL: `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs`
- Query parameters:
  - `id`: `eq.{{ $('Get Latest New Job').item.json.id }}`
- Headers:
  - `apikey`: `[SUPABASE_ANON_KEY]`
  - `Authorization`: `Bearer [SUPABASE_ANON_KEY]`
  - `Content-Type`: `application/json`
  - `Prefer`: `return=representation`
- Body (JSON):
```json
{
  "tech_id": "{{ $('Get Tech by Name').item.json.id }}",
  "status": "Assigned",
  "assigned_at": "{{ new Date().toISOString() }}"
}
```

Expected result: updated job record (Prefer: return=representation returns updated row).

---

### Phase 6 — Confirmation SMS to Dispatcher (AC4)

**Step 6.1 — Code node: Build confirmation message**

Add a Code node after `Assign Job`:

- Node name: `Build Confirmation SMS`
- Type: Code (typeVersion 2)
- Code:
```javascript
const jobId = $('Get Latest New Job').item.json.id;
const jobShort = jobId.slice(-6).toUpperCase();
const techName = $('Get Tech by Name').item.json.name;
const confirmMsg = `Job #${jobShort} assigned to ${techName}.`;
const dispatcherPhone = $('Parse SMS Fields').item.json.from_phone;
const fromNumber = $('Get Client by Phone').item.json.twilio_number;
return [{ json: { confirmation_msg: confirmMsg, dispatcher_phone: dispatcherPhone, from_number: fromNumber } }];
```

**Step 6.2 — Native Twilio node: Send confirmation SMS**

Add a native Twilio node after `Build Confirmation SMS`:

- Node name: `Send Confirmation SMS`
- Type: n8n-nodes-base.twilio (typeVersion 1)
- Credential: Twilio (Account SID + Auth Token)
- Operation: Send SMS
- From: `{{ $('Build Confirmation SMS').item.json.from_number }}`
- To: `{{ $('Build Confirmation SMS').item.json.dispatcher_phone }}`
- Message: `{{ $('Build Confirmation SMS').item.json.confirmation_msg }}`

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC1 | SMS body has extra whitespace | `.trim()` in Extract Tech Name code node |
| AC1 | Name match is case-insensitive | Supabase `ilike` operator handles it |
| AC2 | No client found for that dispatcher_phone | IF client found guard → false branch exits silently |
| AC2 | Multiple techs match (unlikely) | `ilike` returns all — n8n v2 returns first element; first match used |
| AC3 | PATCH fails (network/RLS) | n8n shows error in execution log; no retry (out of scope) |
| AC4 | Job UUID < 6 chars | Impossible — Supabase UUIDs are 36 chars. `.slice(-6)` safe. |
| AC5 | No New job for this client | `status` field will be empty/undefined → IF job is New fails → exits cleanly |
| AC5 | Job status not New | IF node condition fails → exits cleanly (E02S04 handles SMS for this case) |
| AC6 | Supabase native node accidentally used | Implementation constraint — HTTP Request only |
