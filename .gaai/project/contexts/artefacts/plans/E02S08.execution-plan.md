## Implementation Steps

### Step 1 â€” Load Supabase Auth JS client (AC2)

Add `@supabase/supabase-js` CDN script tag in `<head>` before any existing scripts:
```html
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2/dist/umd/supabase.min.js"></script>
```

Initialize Supabase client in the script block (using existing `SUPABASE_URL` / `SUPABASE_ANON_KEY` vars):
```js
var supabaseClient = supabase.createClient(SUPABASE_URL, SUPABASE_ANON_KEY);
```

### Step 2 â€” Add login view HTML (AC1, AC2, AC4, AC7)

Insert a `<div id="login-view">` before `<main class="main-content">`. Contains:
- Card centered on the screen (`--navy-mid` background, `--border` borders)
- Email input (`type="email"`, `font-size: 16px`, `min-height: 52px`)
- Password input (`type="password"`, `font-size: 16px`, `min-height: 52px`)
- Submit button (`min-height: 52px`)
- Error message element (`id="login-error"`) â€” hidden by default; shown on auth failure

Initial state: hidden via `style="display:none"`. JS reveals it when no session.

### Step 3 â€” Wrap existing `<main>` in app view (AC1, AC3, AC6)

Wrap `<main class="main-content">` in `<div id="app-view" style="display:none">`.
Initial state: hidden. JS reveals it when session is active.

### Step 4 â€” Add logout button to nav (AC6)

Add `<button id="logout-btn">` to `.nav-right` in the nav element.
Hidden by default (`style="display:none"`). Shown when session is active.
Styling: matches the existing design system (navy mid background, ice border, Barlow font).

### Step 5 â€” Auth initialization on DOMContentLoaded (AC5)

Replace `document.addEventListener('DOMContentLoaded', ...)` with an auth-aware version:

```js
document.addEventListener('DOMContentLoaded', function() {
  supabaseClient.auth.getSession().then(function(result) {
    if (result.data && result.data.session) {
      showAppView(result.data.session);
    } else {
      showLoginView();
    }
  });
});
```

`showAppView(session)`:
- Hides `#login-view`, shows `#app-view`, shows `#logout-btn`
- Stores `session.access_token` in a module-level variable (`currentAccessToken`)
- Calls `loadTechs()` and `loadJobs()`

`showLoginView()`:
- Hides `#app-view`, hides `#logout-btn`, shows `#login-view`

### Step 6 â€” Login form submit handler (AC2, AC3, AC4)

```js
document.getElementById('login-form').addEventListener('submit', function(e) {
  e.preventDefault();
  var email = document.getElementById('login-email').value;
  var password = document.getElementById('login-password').value;
  var errorEl = document.getElementById('login-error');
  errorEl.style.display = 'none';

  supabaseClient.auth.signInWithPassword({ email: email, password: password })
    .then(function(result) {
      if (result.error) {
        errorEl.textContent = 'Invalid email or password.';
        errorEl.style.display = 'block';
        // AC4: do NOT clear email field
      } else {
        showAppView(result.data.session);
      }
    });
});
```

### Step 7 â€” Logout handler (AC6)

```js
document.getElementById('logout-btn').addEventListener('click', function() {
  supabaseClient.auth.signOut().then(function() {
    currentAccessToken = null;
    showLoginView();
  });
});
```

### Step 8 â€” Thread access token into Supabase REST calls (AC8)

Replace hardcoded `SUPABASE_ANON_KEY` in `Authorization` headers with `currentAccessToken || SUPABASE_ANON_KEY`.
This ensures authenticated requests use the user JWT so RLS `auth.role() = 'authenticated'` policies apply.

Affected fetch calls:
- `loadJobs()` fetch
- `loadTechs()` fetch
- `assignTech()` fetch

## RLS Note (AC8)

The story notes that RLS policies were set for `anon` in Sprint 1 (SELECT/INSERT/UPDATE). The `Authorization: Bearer {access_token}` header in fetch calls will cause Supabase to evaluate policies for `authenticated` role. The note in the story says: "may need to add `auth.role() = 'authenticated'` policies".

This plan adds the auth header to requests but does NOT modify Supabase DB/RLS policies (out of scope for this story â€” it is a Supabase Dashboard configuration action, not a code change). A code comment will note that the dispatcher account must exist in Supabase Auth and that RLS policies must allow authenticated reads/writes.
