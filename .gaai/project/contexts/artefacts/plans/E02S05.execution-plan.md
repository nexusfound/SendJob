## Implementation Steps

### Step 1 — Create output file and directory structure

- Create `/dispatcher-dashboard/index.html` at project root
- This is the deliverable file to be deployed to Netlify

### Step 2 — Design aesthetic direction (frontend-design skill)

- Target audience: dispatcher using on desktop and mobile
- Context: operational tool, high-clarity, fast scanning — not decorative
- Aesthetic direction: **industrial/utilitarian** — clean data table, high readability, functional density
- Design differentiators: monochromatic palette with a single accent color for actions, tight row density on desktop, card-stack on mobile, no unnecessary ornamentation
- Font: distinctive monospace-hybrid for job IDs + clean sans-serif for data (loaded via Google Fonts: `IBM Plex Mono` + `IBM Plex Sans`)

### Step 3 — HTML structure

Define the document skeleton:
- `<head>`: charset, viewport meta (mobile-first), Google Fonts import, inline `<style>` block
- `<body>`: header bar with title + refresh indicator, main content area, two states: `#job-table-container` (table) + `#empty-state` (empty message) + `#error-state` (error message)
- Inline `<script>` block at bottom of body

### Step 4 — CSS: responsive table layout

Requirements derived from AC6 (375px mobile readable, no horizontal scroll):
- Desktop: standard `<table>` layout with all 8 columns visible
- Mobile (≤640px): transform table rows to vertical card stack via CSS (no JS required)
  - Each row becomes a card; `::before` pseudo-elements inject column labels
  - `overflow-x: hidden` on container; `table-layout: fixed` on desktop
- CSS variables for palette: `--bg`, `--surface`, `--border`, `--text-primary`, `--text-secondary`, `--accent`, `--urgency`
- Color palette: off-white background (`#F5F4F1`), charcoal text (`#1A1A1A`), slate border (`#D0CEC9`), burnt-orange accent (`#C45C2D`), muted red urgency (`#B33A3A`)
- Row hover: subtle `#ECEAE5` tint

### Step 5 — JavaScript: data fetch (AC7)

On `DOMContentLoaded`:

```
const SUPABASE_URL = 'https://tmoxppuxrwlabqzfzvin.supabase.co'
const SUPABASE_ANON_KEY = 'SUPABASE_ANON_KEY_PLACEHOLDER'
```

Fetch pattern (HTTP Request, not SDK — per conventions.md):
```
fetch(`${SUPABASE_URL}/rest/v1/jobs?select=id,customer_name,address,service_type,is_urgent,status,tech_name,created_at&order=created_at.asc`, {
  headers: {
    'apikey': SUPABASE_ANON_KEY,
    'Authorization': `Bearer ${SUPABASE_ANON_KEY}`
  }
})
```

- `order=created_at.asc` satisfies AC2 (oldest first)
- No `limit` parameter — all records returned (AC4)

### Step 6 — JavaScript: render logic

After fetch resolves:
- If response not ok: show `#error-state` with error message (AC8)
- Parse JSON array
- If `data.length === 0`: show `#empty-state` (AC5)
- Otherwise: build table rows, hide `#empty-state`, show `#job-table-container`

Row rendering per AC1:
| Column | Source field | Fallback |
|--------|-------------|---------|
| Job ID | `job.id` | — |
| Customer | `job.customer_name` | — |
| Address | `job.address` | — |
| Service Type | `job.service_type` | — |
| Urgency | `job.is_urgent ? 'URGENT' : 'Normal'` | — |
| Status | `job.status` | — |
| Assigned Tech | `job.tech_name \|\| 'Unassigned'` | "Unassigned" (AC3) |
| Submitted | formatted `job.created_at` | — |

### Step 7 — JavaScript: error and empty states

- `#error-state`: visible when fetch fails; message: "Unable to load jobs. Check your connection and try again."
- `#empty-state`: visible when array is empty; message: "No jobs in queue."
- Both hidden by default via `display: none`

### Step 8 — Time formatting

Use `Intl.DateTimeFormat` for human-readable time display:
```javascript
function formatTime(isoString) {
  return new Intl.DateTimeFormat('en-US', {
    month: 'short', day: 'numeric',
    hour: 'numeric', minute: '2-digit'
  }).format(new Date(isoString))
}
```

## Risk Notes

- `SUPABASE_ANON_KEY_PLACEHOLDER` is a placeholder in the file — the human must replace this with the real anon key from the Supabase dashboard before deploying. This is intentional: embedding secrets in a deliverable artefact is a rule violation.
- `tech_name` field: the Story notes say "tech_name (or join to techs table)". A join would require a PostgREST foreign key relation. Using a direct `tech_name` column is the simpler, safer approach for this story. If `jobs` table has no `tech_name` column, a join to `techs` via `tech_id` would be needed. This risk is noted in the impl-report.
- Mobile layout: the CSS card-stack approach (no JS) is the most reliable for cross-browser compatibility.
