# Execution Plan — E03S08: Tech Dashboard Expired/Invalid Token Error Page

## High-Level Approach

This story modifies one file: `tech-dashboard/index.html`. The error state structure already exists in the file (a generic error div at line 521–528). The work is to:

1. Differentiate the two error types (expired/invalid token vs. network failure) with separate messages.
2. Ensure the expired/invalid path uses the specific AC3 message.
3. Ensure the network-failure path uses the AC6 generic message.
4. Confirm no partial dashboard content is visible when error is shown (AC5).
5. Confirm styling matches the dark navy design system (AC4) — the existing `.error-state` CSS already satisfies this.

Approach evaluation: **Skipped.** The error state HTML and CSS patterns are already established in the file. No new technology, library, or architectural decision required.

## Implementation Sequence

### Step 1 — HTML: Split error state into two named variants

**Target:** Replace the single generic `#error-state` div with two distinct error divs:
- `#error-token` — shown for missing/invalid token (AC1, AC2, AC3)
- `#error-network` — shown for network failure on token lookup (AC6)

Both styled with existing `.error-state` class (AC4).

**Edge cases:**
- Both divs must be `display:none` by default
- Neither the `#dashboard-content` nor any job list content may be visible when either error is shown (AC5)

### Step 2 — CSS: No new styles needed

Existing `.error-state`, `.error-title`, `.error-sub`, `.error-icon` cover both variants. No CSS changes required.

### Step 3 — JS: Update `showError()` and `init()` to use distinct error states

**Current behaviour:**
- `showError()` is a single function that shows `#error-state`
- `init()` calls `showError()` for both "no token" and "empty result" and "fetch error"

**New behaviour:**
- `showTokenError()` — shows `#error-token`, sets status dot to error
- `showNetworkError()` — shows `#error-network`, sets status dot to error
- Call `showTokenError()` for: no token in URL (AC1), empty result from Supabase (AC2)
- Call `showNetworkError()` for: catch block during token fetch (network failure) (AC6)

**AC3 message:** "This link has expired. Text JOBS to your dispatcher to get a new one."

Note per Story Notes: when the token is invalid, `client_id` is unknown, so we cannot retrieve the client's Twilio number. Use the generic instruction "contact your dispatcher" rather than a specific number. This satisfies AC3's intent while remaining technically correct given the constraint noted in the story.

**AC5 invariant:** `#dashboard-content` starts hidden (`display:none`) and is only revealed after successful token validation. Both error paths return early and never set `$dashboard.style.display = 'block'`.

### Step 4 — Self-validation against ACs

Before committing, verify each AC is satisfied by code inspection:

| AC | Verification |
|----|-------------|
| AC1 | `if (!token)` branch calls `showTokenError()` before any fetch |
| AC2 | `if (techs.length === 0)` branch calls `showTokenError()` |
| AC3 | `#error-token` div contains exact message text |
| AC4 | Both error divs use `.error-state` class (dark navy card, not browser default) |
| AC5 | `#dashboard-content` never made visible in error paths; no partial content shown |
| AC6 | `catch(err)` in token fetch calls `showNetworkError()` with generic message |

## Test Checkpoints

1. **Token missing:** Remove `?token=` from URL → token error div visible, dashboard hidden
2. **Token invalid:** Pass `?token=invalid-value` → Supabase returns empty array → token error visible
3. **Token valid:** Pass a real token → error divs hidden, dashboard renders normally
4. **Network error simulation:** Cannot test in static HTML without mocking — verified by code path inspection (catch block → `showNetworkError()`)
5. **Partial content check:** In all error paths, confirm `#dashboard-content` is never set to `display:block`
