# E03S05 Execution Plan — Tech Dashboard: En Route and On Site Action Buttons

## Summary

Add En Route and On Site action buttons to each job card in `tech-dashboard/index.html`.
Buttons send a PATCH to Supabase `jobs` table using the anon key (no JWT — existing RLS anon UPDATE policy permits this).
After a successful PATCH, the card's button area updates in place (no page reload).
On failure, show an inline error and re-enable the button.
During the request, disable the button to prevent double-submission.

---

## Implementation Sequence

### Step 1 — Add CSS for action buttons

Add the following CSS blocks to the `<style>` section, after the `.job-count-badge` rule:

1. `.job-card-actions` — full-width container at the bottom of each card body
2. `.action-btn` — base button: full width, min-height 52px (exceeds 48px AC6 requirement), Barlow Condensed, border-radius 10px, font-size 16px, font-weight 700
3. `.action-btn.btn-en-route` — amber color: `background: rgba(255,184,0,0.15)`, `color: var(--warn)`, `border: 1px solid rgba(255,184,0,0.5)`, hover/active states
4. `.action-btn.btn-on-site` — green color: `background: rgba(0,214,143,0.15)`, `color: var(--success)`, `border: 1px solid rgba(0,214,143,0.5)`, hover/active states
5. `.action-btn:disabled` — muted opacity (0.5), cursor not-allowed
6. `.card-error` — inline error below button: font-size 13px, color var(--urgent), padding 6px 16px, visible when error occurs

### Step 2 — Add `patchJobStatus(jobId, newStatus, cardEl)` function

New async function in the `<script>` block. Responsibilities:
- Accept `jobId` (UUID), `newStatus` ('En Route' | 'On Site'), `cardEl` (reference to the job card DOM element)
- Disable all `.action-btn` elements inside `cardEl`
- Clear any existing `.card-error` message inside `cardEl`
- Perform `fetch(SUPABASE_URL + '/rest/v1/jobs?id=eq.' + encodeURIComponent(jobId), { method: 'PATCH', headers: { ...HEADERS, 'Content-Type': 'application/json', 'Prefer': 'return=minimal' }, body: JSON.stringify({ status: newStatus }) })`
- On success (res.ok): call `updateCardButtons(cardEl, newStatus)` to swap buttons
- On failure: re-enable button(s), show `.card-error` with message "Update failed — tap to retry"

### Step 3 — Add `updateCardButtons(cardEl, newStatus)` function

Replaces the `.job-card-actions` section of a card DOM element after a successful PATCH:
- If `newStatus === 'En Route'` → show On Site button (status just became En Route → next action is On Site)
- If `newStatus === 'On Site'` → remove the actions container entirely (AC3: no further buttons when On Site)
- Update the status badge text and class on the card header

### Step 4 — Modify `renderJobs()` to append action button area

Inside the `jobs.forEach` loop in `renderJobs()`, after building the card HTML, append a `.job-card-actions` section:
- If `job.status === 'Assigned'` → render En Route button with `data-job-id` attribute and `data-next-status="En Route"`
- If `job.status === 'En Route'` → render On Site button with `data-job-id` attribute and `data-next-status="On Site"`
- If `job.status === 'On Site'` → no action section (AC3)

Button renders as:
```html
<div class="job-card-actions">
  <button class="action-btn btn-en-route" data-job-id="{uuid}" data-next-status="En Route">
    🚗 En Route
  </button>
</div>
```

### Step 5 — Attach event listeners via event delegation

After `$jobsList.appendChild(card)`, use a click listener on the button inside the card:
- `card.querySelector('.action-btn').addEventListener('click', ...)` — calls `patchJobStatus(jobId, nextStatus, card)`
- This avoids polluting the global scope with per-button listener references

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC6 | Button too small on iOS | `min-height: 52px`, `width: 100%`, `padding: 14px` — exceeds 48px |
| AC7 | PATCH returns 200 but status doesn't change | Trust res.ok — Supabase 200 = success on PATCH with `Prefer: return=minimal` |
| AC8 | Network timeout or non-ok response | Show `.card-error`, re-enable button |
| AC9 | Double-tap before response | Button disabled immediately on first tap |
| AC3 | Tech refreshes page after On Site | Jobs fetch already filters `status=in.(Assigned,En Route,On Site)` — On Site jobs show, no buttons |
| AC4/5 | jobId is a UUID with hyphens | `encodeURIComponent()` handles special chars in query param |
| AC7 | Rapid status skip (Assigned → On Site) | Not possible — each status only shows one button at a time |

## Test Checkpoints

1. **TC1 — Assigned card shows En Route button only:** Load dashboard for a tech with an Assigned job. Verify En Route button is visible, On Site button is NOT visible.
2. **TC2 — En Route card shows On Site button only:** Load with an En Route job. Verify On Site button visible, En Route NOT visible.
3. **TC3 — On Site card shows no buttons:** Load with On Site job. No action buttons visible.
4. **TC4 — Button is ≥48px tap target:** Inspect rendered button height in browser dev tools. Must be ≥48px.
5. **TC5 — Tap En Route disables button, patches Supabase:** Click En Route. Button disables. Network tab shows PATCH to /rest/v1/jobs?id=eq.{uuid} with `{ status: 'En Route' }`.
6. **TC6 — After successful PATCH, card updates:** After PATCH 200, card header badge changes to En Route, button area swaps to On Site button.
7. **TC7 — Tap On Site removes buttons:** Click On Site. After PATCH success, no buttons in card.
8. **TC8 — On PATCH failure, inline error shown, button re-enabled:** Simulate network failure (offline / bad URL). Inline error appears on card. Button becomes tappable again.
9. **TC9 — Double-tap prevention:** Click button rapidly twice. Only one PATCH request fires.

## Risk Register

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Supabase RLS blocks anon PATCH | Low | Story notes confirm anon UPDATE policy already covers `jobs` table |
| Button renders but job.id is undefined | Low | `data-job-id` uses `job.id` (UUID) not `formatJobId()` (last-6 display string) |
| Button min-height not honoured by iOS Safari | Low | Use `min-height` + `padding` combined; iOS respects padded height |
