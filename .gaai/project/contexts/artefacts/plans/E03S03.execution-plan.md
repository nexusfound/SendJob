# E03S03 Execution Plan ‚Äî N8N: Tech Notification SMS with Magic Link

## Summary

Produce a single importable n8n workflow JSON (`n8n-workflows/E03S03-tech-notification-sms.json`) that, when triggered after E03S02 token generation, performs the following sequence:

1. Receive trigger payload (tech_id, job_id, token, client_id ‚Äî from E03S02 context or webhook)
2. Fetch job details from Supabase `jobs` table (customer_name, service_address, customer_phone, issue_description, client_id)
3. Fetch client record from Supabase `clients` table (business_name, twilio_number, client_id)
4. Fetch tech record from Supabase `techs` table (phone, contact_method, magic_link_token, name)
5. Check tech `contact_method`:
   - If `sms` ‚Üí send via native n8n Twilio node
   - If `email` ‚Üí stub (log only ‚Äî acceptable per AC6)
6. Build canonical SMS body with opt-out language check
7. Send SMS via Twilio native node

## Architecture Decision: Trigger Mode

The E03S02 workflow ends at the "Token PATCH Succeeded?" IF node. E03S03 must be triggered after the token is stored. Two approaches:

**Option A ‚Äî Webhook trigger (standalone workflow):** E03S02's true branch calls a webhook that triggers E03S03. Clean separation; matches E03S02's integration note for "MODE 1 standalone."

**Option B ‚Äî Inline nodes appended to E03S02:** No webhook round-trip; single workflow. However E03S02 is already delivered (status: done) and modifying it is out of scope.

**Decision: Option A ‚Äî Webhook trigger.** E03S03 is a standalone workflow. Expects POST body: `{ tech_id, job_id, token, client_id }`. This is consistent with how E03S02 was structured and avoids modifying the already-delivered E03S02.

The E03S02 workflow comment already documents that Mode 1 (webhook) is available. Brian wires E03S02 ‚Üí E03S03 via HTTP Request node after the "Respond to Webhook" node, or appends a call to the E03S03 webhook in E03S02's true branch.

## Node Sequence

```
1. Trigger: Tech SMS Webhook          (Webhook, typeVersion 2, POST /sendjob-tech-sms)
2. GET Job Details                    (HTTP Request, typeVersion 4.2 ‚Äî GET /rest/v1/jobs?id=eq.[job_id])
3. GET Client Details                 (HTTP Request, typeVersion 4.2 ‚Äî GET /rest/v1/clients?id=eq.[client_id])
4. GET Tech Details                   (HTTP Request, typeVersion 4.2 ‚Äî GET /rest/v1/techs?id=eq.[tech_id])
5. Build SMS Body                     (Code, typeVersion 2 ‚Äî assemble message + opt-out check)
6. Contact Method: SMS or Email?      (IF ‚Äî $('GET Tech Details').item.json.contact_method === 'sms')
7a. [True]  Send Tech SMS             (Twilio native node)
7b. [False] Log: Email Stub           (Code, typeVersion 2 ‚Äî stub log, AC6)
8.  Respond to Webhook                (Respond to Webhook)
```

## Step Detail

### Step 1 ‚Äî Trigger: Tech SMS Webhook
- Path: `sendjob-tech-sms`
- Expects POST body: `{ "tech_id": "<uuid>", "job_id": "<uuid>", "token": "<uuid>", "client_id": "<uuid>" }`
- `responseMode`: `responseNode`

### Step 2 ‚Äî GET Job Details
- URL: `=https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs?id=eq.{{ $('Trigger: Tech SMS Webhook').item.json.body.job_id }}&select=customer_name,customer_phone,service_address,issue_description,client_id`
- Headers: `apikey`, `Authorization: Bearer`, `Content-Type: application/json`
- Method: GET, typeVersion: 4.2

### Step 3 ‚Äî GET Client Details
- URL: `=https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/clients?id=eq.{{ $('Trigger: Tech SMS Webhook').item.json.body.client_id }}&select=business_name,twilio_number`
- Headers: same as above

### Step 4 ‚Äî GET Tech Details
- URL: `=https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/techs?id=eq.{{ $('Trigger: Tech SMS Webhook').item.json.body.tech_id }}&select=phone,contact_method,magic_link_token,name`
- Headers: same as above

### Step 5 ‚Äî Build SMS Body (Code node)
Assembles:
- `businessName` from `$('GET Client Details').item.json.business_name`
- `customerName` from `$('GET Job Details').item.json.customer_name`
- `serviceAddress` from `$('GET Job Details').item.json.service_address`
- `customerPhone` from `$('GET Job Details').item.json.customer_phone`
- `issueDescription` from `$('GET Job Details').item.json.issue_description`
- `token` from `$('Trigger: Tech SMS Webhook').item.json.body.token`
- `magicLinkUrl` = `https://tech.sendjob.app?token=[token]`
- Canonical template: `[Business Name] - New job assigned: [Customer], [Address], [Phone]. Issue: [Description]. View job: [URL]`
- `isFirstSms` check: if `$('GET Tech Details').item.json.magic_link_token` was previously null/empty before this assignment (not determinable at this point ‚Äî default: append opt-out). **Per story notes: append opt-out on first job for this tech.** Simplest safe approach: always append `Reply STOP to unsubscribe` (over-inclusive is safe; under-inclusive risks compliance failure).
- Output: `{ smsBody, techPhone, contactMethod, businessName }`

### Step 6 ‚Äî IF: contact_method === 'sms'
- Condition: `$('GET Tech Details').item.json.contact_method` === `sms`
- True ‚Üí Step 7a (Twilio)
- False ‚Üí Step 7b (Email stub)

### Step 7a ‚Äî Send Tech SMS (native Twilio node)
- `from`: `$('GET Client Details').item.json.twilio_number` (the client's 10DLC number)
- `to`: `$('GET Tech Details').item.json.phone`
- `message`: `$('Build SMS Body').item.json.smsBody`
- Credential: Twilio (Account SID + Auth Token)

### Step 7b ‚Äî Log: Email Stub (Code node)
- Log: `console.log('[SendJob] E03S03: email path not yet implemented for tech_id=' + ...)`
- Return stub result for AC6

### Step 8 ‚Äî Respond to Webhook
- respondWith: `allIncomingItems`
