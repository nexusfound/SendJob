# E02S07 Execution Plan — Dashboard: Dropdown Assignment

## Approach Rationale

Single-file HTML modification (`dispatcher-dashboard/index.html`). The Supabase PATCH pattern is explicitly defined in the Story notes and mirrors the existing fetch pattern established in E02S05. No new library, service, or architectural decision required. Approach evaluation not invoked — convention is established and documented.

---

## Implementation Sequence

### Step 1 — Load techs list on page initialisation

**Target:** JavaScript `<script>` block in `index.html`
**Action:** Add a `loadTechs()` function that fetches from:
```
GET /rest/v1/techs?client_id=eq.{CLIENT_ID}&select=id,name
```
- Uses same `apikey` / `Authorization: Bearer` headers as `loadJobs()`
- Stores result in a module-level `var techsList = []` array
- Called once on `DOMContentLoaded`, before or alongside `loadJobs()`

**Client ID:** Story notes say "hardcode the test client UUID". The existing dashboard uses `SUPABASE_URL` and `SUPABASE_ANON_KEY` as module-level constants — follow the same pattern. Add `var CLIENT_ID = 'REPLACE_WITH_YOUR_CLIENT_UUID';` near the top of the config block. This is intentional: the human must substitute the real UUID before going live.

**Edge cases:**
- Empty techs array → dropdown shows only a disabled "— Select Tech —" placeholder; no crash
- Fetch error → log to console; `techsList` stays empty; assignment UI gracefully degrades (dropdowns disabled)

### Step 4 — Modify `buildJobRow()` to inject dropdown or read-only tech display

**Target:** `buildJobRow(job)` function
**Action:**

Determine assignment eligibility:
```javascript
var isNew = job.status && job.status.toLowerCase() === 'new';
```

**If `isNew`:**
- Build a `<select class="assign-select">` with:
  - First option: `<option value="">— Select Tech —</option>` (disabled, selected)
  - One `<option value="{tech.id}">{tech.name}</option>` per entry in `techsList`
- Attach an `onchange` handler that calls `assignTech(job.id, selectEl)` (defined in Step 5)

**If NOT `isNew` (Assigned / En Route / On Site / Complete):**
- Render the assigned tech name as read-only text (existing `assignedTech` variable)

**Tech column:** Keep existing logic (shows "Unassigned" span for new jobs, tech name for assigned).

**New Assign column cell:**

### Step 5 — Implement `assignTech(jobId, selectEl)` function

**Target:** JavaScript `<script>` block
**Action:**

```
function assignTech(jobId, selectEl) {
  var techId = selectEl.value;
  if (!techId) return;  // guard: placeholder selected

  selectEl.disabled = true;  // prevent double-submit

  var endpoint = SUPABASE_URL + '/rest/v1/jobs?id=eq.' + jobId;
  var payload = {
    tech_id:     techId,
    status:      'Assigned',
    assigned_at: new Date().toISOString()
  };

  fetch(endpoint, {
    method: 'PATCH',
    headers: {
      'apikey':        SUPABASE_ANON_KEY,
      'Authorization': 'Bearer ' + SUPABASE_ANON_KEY,
      'Content-Type':  'application/json',
      'Prefer':        'return=representation'
    },
    body: JSON.stringify(payload)
  })
  .then(function(response) {
    if (!response.ok) {
      return response.text().then(function(body) {
        throw new Error('PATCH failed ' + response.status + ': ' + (body || 'no detail'));
      });
    }
    return response.json();
  })
  .then(function(data) {
    // AC3: update row in-place without full reload
    var techName = '';
    for (var i = 0; i < techsList.length; i++) {
      if (techsList[i].id === techId) { techName = techsList[i].name; break; }
    }
    updateRowAfterAssign(selectEl, techName, jobId);
  })
  .catch(function(err) {
    // AC6: show inline error; re-enable select
    selectEl.disabled = false;
    showAssignError(selectEl, err.message || 'Assignment failed. Try again.');
  });
}
```

### Step 8 — Wire `loadTechs()` into `DOMContentLoaded`

**Target:** `document.addEventListener('DOMContentLoaded', ...)` call at bottom of script
**Action:** Change from `loadJobs` alone to calling both:
```javascript
document.addEventListener('DOMContentLoaded', function() {
  loadTechs();
  loadJobs();
});
```

`loadTechs()` populates `techsList` before `loadJobs()` renders rows. Both are async — in practice techs may finish after jobs render on very fast connections. The guard in Step 4 (empty techsList → disabled select) handles this gracefully.

## Edge Cases & Test Checkpoints

| AC | Checkpoint | Pass condition |
|----|-----------|----------------|
| AC1 | New-status row | Shows populated `<select>` with tech options |
| AC1 | techs fetch error | Select shows "No techs loaded" and is disabled |
| AC2 | Dispatcher selects tech | PATCH fires with correct `tech_id`, `status=Assigned`, `assigned_at=ISO` |
| AC2 | Headers | `apikey`, `Authorization: Bearer`, `Content-Type: application/json`, `Prefer: return=representation` present |
| AC3 | After successful PATCH | Row updates in-place: status badge → Assigned, tech cell → tech name, dropdown → read-only |
| AC4 | Non-New status row | No dropdown rendered — read-only tech name only |
| AC5 | Mobile 375px | `<select>` min-height ≥ 52px, font-size = 16px |
| AC6 | PATCH fails (4xx/5xx) | Inline error shown; row state not silently changed |
| AC7 | PATCH request | Uses `apikey` + `Authorization: Bearer` headers (same pattern as existing fetch) |
