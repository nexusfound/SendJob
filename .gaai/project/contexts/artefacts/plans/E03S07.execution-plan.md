# Execution Plan — E03S07: N8N On Site Customer SMS + Status Update

## High-Level Approach

This story delivers an n8n polling workflow that mirrors E03S06 exactly with three changes:
1. Status filter: `On Site` (instead of `En Route`)
2. Sentinel column: `on_site_sms_sent` (instead of `en_route_sms_sent`)
3. SMS template: `[Business Name]: Your technician has arrived.` (instead of "on the way")

The workflow:
1. Polls the Supabase `jobs` table every 30 seconds for rows where `status = 'On Site'` AND `on_site_sms_sent = false`
2. For each matching row: fetches client `business_name` from `clients`, sends customer SMS via native Twilio node
3. Sets `on_site_sms_sent = true` on the job to prevent duplicate sends
4. Is tolerant of null `customer_phone` (AC5) — skips SMS, does not error
5. SMS failure does NOT revert job status (AC6)

**Twilio note (user context):** Twilio A2P campaign not yet approved. The `From` number in the Twilio node uses `FROM_NUMBER_PLACEHOLDER` — human replaces with approved 10DLC number before activating.

Approach evaluation: **Skipped.** Exact mirror of E03S06 — all patterns established, no alternatives to evaluate. Schema migration is a simple additive boolean column.

## Implementation Sequence

### Step 1 — Schema Migration: Add `on_site_sms_sent` to `jobs`

Add boolean column `on_site_sms_sent` with default `false` to the `jobs` table.

This column is the deduplication sentinel. The polling filter is:
```
status=eq.On Site&on_site_sms_sent=eq.false
```

After SMS send (or skip on null phone), the workflow PATCHes `on_site_sms_sent = true`.

**Migration file:** `supabase/migrations/E03S07-add-on-site-sms-sent.sql`
```sql
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS on_site_sms_sent boolean DEFAULT false;
```

**Edge cases:**
- `IF NOT EXISTS` guard prevents failure if column already added manually
- Default `false` ensures all existing rows start unpolluted
- Column is nullable-safe (workflow uses `eq.false` filter, not `is.null`)

**Note:** `jobs` table anon UPDATE policy already exists (confirmed E03S05 + E03S06) — no new RLS policy needed.

### Step 2 — N8N Workflow: Polling Trigger

Node: **Schedule Trigger** (n8n-nodes-base.scheduleTrigger, typeVersion 1.2)
- Interval: every 30 seconds
- Same as E03S06

### Step 3 — N8N Workflow: GET Jobs (On Site, SMS Not Sent)

Node: **GET Jobs Pending On Site SMS** (HTTP Request)
- GET `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs?status=eq.On%20Site&on_site_sms_sent=eq.false&select=id,client_id,customer_name,customer_phone`
- Headers: `apikey` + `Authorization: Bearer` (SUPABASE_ANON_KEY_PLACEHOLDER)
- Returns array of matching jobs (may be empty — handled by Split In Items)

**URL encoding:** `On Site` encoded as `On%20Site`.

### Step 4 — N8N Workflow: Split Items (Loop per Job)

Node: **Split In Items** (n8n-nodes-base.splitInBatches, typeVersion 3, batchSize: 1)

**Known import issue:** After JSON import into n8n, Split In Items will wire to the `done` output — must manually rewire to `loop` output before activating.

### Step 5 — N8N Workflow: GET Client Details

Node: **GET Client Details** (HTTP Request)
- GET `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/clients?id=eq.{{ $('Split In Items').item.json.client_id }}&select=business_name,twilio_number`
- Headers: `apikey` + `Authorization: Bearer` (SUPABASE_ANON_KEY_PLACEHOLDER)

### Step 6 — N8N Workflow: Check customer_phone

Node: **Customer Phone Present?** (IF node)
- Condition: `$('Split In Items').item.json.customer_phone` is not empty

**AC5:** If `customer_phone` is null/empty → false branch → skip SMS, go directly to Mark SMS Sent.

### Step 7 — N8N Workflow: Build SMS Body

Node: **Build SMS Body** (Code node)
- Assembles: `[Business Name]: Your technician has arrived.`
- Pulls `business_name` from GET Client Details output
- Validates required fields — throws on missing `business_name`
- Returns: `{ smsBody, customerPhone, twilioFromNumber }`

**AC2:** SMS text exactly: `[Business Name]: Your technician has arrived.`

### Step 8 — N8N Workflow: Send Customer SMS (Twilio native node)

Node: **Send Customer SMS** (n8n-nodes-base.twilio, typeVersion 1)
- `from`: `FROM_NUMBER_PLACEHOLDER` (human replaces with approved 10DLC number)
- `to`: `$('Build SMS Body').item.json.customerPhone`
- `message`: `$('Build SMS Body').item.json.smsBody`
- `continueOnFail: true` — SMS failure must not error the workflow (AC6)

**AC4:** Native Twilio node — NOT HTTP Request (conventions.md, DEC-006).

### Step 9 — N8N Workflow: PATCH jobs — Mark SMS Sent

Node: **PATCH jobs — Mark On Site SMS Sent** (HTTP Request)
- PATCH `https://tmoxppuxrwlabqzfzvin.supabase.co/rest/v1/jobs?id=eq.{{ $('Split In Items').item.json.id }}`
- Body: `{ "on_site_sms_sent": true }`
- Headers: `apikey`, `Authorization: Bearer`, `Content-Type: application/json`, `Prefer: return=representation`
- `continueOnFail: true`

**AC1:** Status remains `On Site` — this node only writes `on_site_sms_sent = true`, never touches `status` field.

Both the SMS-sent path AND the phone-null skip path merge into this PATCH node.

### Step 10 — Self-validation against ACs

| AC | Verification |
|----|-------------|
| AC1 | Poll filter `status=eq.On%20Site` + PATCH only writes `on_site_sms_sent`; `status` field never touched |
| AC2 | Code node assembles exact template: `[Business Name]: Your technician has arrived.` |
| AC3 | `business_name` read from `clients?id=eq.[client_id]` — never hardcoded |
| AC4 | Twilio node type `n8n-nodes-base.twilio` used (not httpRequest) |
| AC5 | IF node checks `customer_phone`; false branch skips Twilio node entirely |
| AC6 | `continueOnFail: true` on Twilio node; PATCH only writes `on_site_sms_sent`, not `status` |

## Test Checkpoints

1. **Polling fires, no eligible jobs:** schedule fires, GET returns empty array — workflow completes silently
2. **Job eligible, phone present:** polling finds job, GET client returns business_name, SMS sent (or Twilio credential error with `continueOnFail:true`), PATCH marks sent — job no longer polled next cycle
3. **Job eligible, phone null:** polling finds job, IF node routes to false branch, PATCH marks sent — no SMS, no error
4. **Duplicate guard:** after first cycle marks `on_site_sms_sent = true`, same job does not appear in subsequent polls
5. **Business name lookup:** verify `business_name` in SMS matches `clients.business_name` — never hardcoded
