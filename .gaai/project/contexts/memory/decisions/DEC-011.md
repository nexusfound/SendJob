# DEC-011 — Parameterized n8n Workflows — One Set Handles All Clients

## Context

SendJob serves multiple clients (trades businesses). A naive approach would duplicate workflows per client. This creates maintenance overhead and drift. The correct architecture uses a single parameterized workflow set that looks up client config at runtime.

## Decision

n8n workflows are parameterized — one set of workflows handles all clients. Client configuration (business name, dispatcher phone, urgency keywords, payment link, Twilio number) is looked up from the `clients` Supabase table by `subdomain` on every webhook trigger. Never build per-client workflow copies.

## Impact

- Every workflow starts with a Supabase lookup: `GET /rest/v1/clients?subdomain=eq.{subdomain}`
- Subdomain is passed in the request body as field `subdomain` — NOT Host header
- New clients are onboarded by inserting a row into `clients` table — no workflow changes needed
- All SMS messages use `clients.business_name` — never hardcode client names in workflow nodes
